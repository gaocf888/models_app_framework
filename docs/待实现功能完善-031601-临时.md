1. LangChain / LangGraph 编排层还未真正接入
涉及 TODO：2, 3, 4, 6, 7, 11, 12

当前情况：

已有：
LLMClient（vLLM HTTP）、配置中心、RAGService、ConversationManager、ChatbotService、AnalysisService、NL2SQLChain 等。
仍是占位/未实现部分：
没有真正的 LangChain/LangGraph 链：
Chatbot 目前是 ChatbotService 直接拼占位回答，没有用 LangChain 的 Chain / Agent 结构。
综合分析 AnalysisService 也是占位逻辑，没有多步 Agent 流程。
NL2SQL 虽然有 NL2SQLChain，但也是“自己写的链条”，未通过 LangChain 的 Runnable/Chain 体系组织。
PromptTemplateRegistry 没有真正被各链路使用：
只实现了 PromptTemplateRegistry 和 configs/prompts.yaml，但 Chatbot/Analysis/NL2SQL 的 prompt 构造仍是内嵌在各服务/PromptBuilder 里，没有统一从 Registry 取模板 + 版本信息。
没有接入 LangSmith：
文档中规划了用 LangSmith 做链路观测与 A/B 实验记录，但目前代码没有任何 LangSmith SDK 的集成点。
要达生产级别，需要：

为 Chatbot、Analysis、NL2SQL 分别实现基于 LangChain 的 Chain/Agent，并在其中使用：
LLMClient 作为 LLM backend；
PromptTemplateRegistry 来选择 Prompt 版本（支持 A/B）；
RAGService/ConversationManager 作为 LangChain 工具或 Memory。
在关键链路上增加 LangSmith trace 上报（可配置开关）。

文档/Schema/业务知识/问答样例的摄入流程；
/rag/ingest、/rag/datasets 等管理接口；
与 SchemaMetadataService 联动（Schema 变更自动/手动重新索引）。
