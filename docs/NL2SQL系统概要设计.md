## 一、目标与范围

本系统旨在基于 Python 和大语言模型（LLM），构建一套可在企业环境中落地的 NL2SQL（自然语言转 SQL）系统，使非技术人员可以通过自然语言直接查询业务数据库，实现“所想即所得”的数据访问能力。

系统首期聚焦于：
- **查询类只读 SQL**（SELECT），面向典型业务分析场景；
- 支持 **中文自然语言问题**，生成 MySQL / PostgreSQL 等主流数据库方言的 SQL；
- 支持单库多表的分析查询，并为后续多轮对话、复杂报表和领域微调预留扩展空间。

---

## 二、总体架构设计

整体采用 **“LLM + RAG + 数据库适配层 + 应用接入层”** 的分层架构，将传统多模块流水线（NER、意图识别、句法分析等）收敛为以大模型为核心的智能层，由周边子系统保障可控性与可靠性。

从数据流看，典型链路为：

> 自然语言问题 → Schema 语义检索（RAG） → Prompt 编排 → LLM 生成候选 SQL → SQL 校验与自我修正 → 数据库执行 → 结果展示与解释

从系统组件看，划分为三大层：

1. **接入层（API & UI 层，基于 FastAPI 实现）**
   - 使用 `FastAPI` 作为统一接入框架，提供 HTTP/REST 或 WebSocket 接口，对接 Web 前端、BI 工具或内部系统。
   - 负责用户认证、请求路由、会话管理以及结果展示，可按路由划分查询、Schema 管理、运维监控等接口分组。
   - **在本基座中的落地**：NL2SQL 与 RAG 同属 **AI 应用基础能力**；NL2SQL **既** 可通过 **`POST /nl2sql/query`** 被 BI、脚本等 **直接调用**，**又** 可被智能客服等场景 **内嵌复用**（结构化问数意图），详见 `docs/大小模型应用技术架构与实现方案.md` §1、§4.6 与 `enterprise-level_transformation_docs/企业级NL2SQL实现方案.md`。

2. **NL2SQL 智能层（核心逻辑，Python 实现）**
   - **Schema 管理与向量检索子系统（RAG）**：维护数据库模式的语义索引，并在查询时检索最相关的表与字段。
   - **Prompt 编排器**：根据用户问题、检索到的 Schema 信息和对话上下文动态构造 Prompt。
   - **LLM 调用器**：统一封装对 Qwen3 / GPT / DeepSeek 等大模型的调用逻辑。
   - **SQL 校验与自我修正模块**：对生成 SQL 做语法与安全校验，基于执行反馈驱动模型自我修正。

3. **数据访问层**
   - **数据库 Schema 管理模块**：从业务数据库自动拉取表、字段、注释等信息并保持同步。
   - **多数据源适配器**：屏蔽 MySQL / PostgreSQL 等不同数据库的差异。
   - **SQL 执行与安全控制模块**：负责实际执行 SQL，并对敏感操作、超长查询等进行防护。

## 三、关键模块设计

### 3.1 Schema 管理与向量检索（RAG 子系统）

**目标**：将用户口中的“产品”“销售额”“2023 年”等概念，可靠映射到特定数据库中的 `table.column`，并结合业务语义和高质量示例，为大模型提供“开卷考试式”的上下文，是 NL2SQL 能否正确落地的关键。

**知识库构成**（来自数据库设计文档及补充资料）：
- **核心元数据（Schema / DDL）**：表名、列名、数据类型、主外键关系、列注释等；
- **增强的业务语义**：业务定义（如“高价值客户”阈值）、专业术语解释（SKU、ARPU 等）、常用统计口径说明；
- **高质量问答对**：领域内高频的“自然语言问题 + 标准 SQL”样例，用于作为 Few-shot 参考；
- **可选示例数据**：典型表的若干行示例记录，帮助模型理解字段含义和取值分布。

上述内容会被切分为合适粒度的“文本块”（例如单字段描述、单条业务规则、单个问答对），统一向量化后存入向量数据库。

**向量化与存储**：
- 使用嵌入模型（如 `qwen3-embedding`、`bge-large-zh` 等）对 Schema 片段、业务规则、问答对等文本块进行向量化；
- 采用向量检索库进行存储和检索，并按“命名空间/类型”区分不同内容：
  - `schema`：表/字段 DDL 与注释；
  - `biz_knowledge`：业务定义、口径说明、术语解释；
  - `qa_examples`：典型问答对；
  - `sample_rows`：样例数据说明。
  - PoC 阶段可使用 **FAISS + 本地存储（如 SQLite / 文件）**；
  - 生产阶段可升级为 **Milvus / pgvector / Elasticsearch dense vector** 等。

**RAG 检索流程（问题 → 上下文）**：
1. **用户提问与编码**：接收自然语言问题，将问题文本通过嵌入模型编码为向量；
2. **多通道检索**：在上述多个命名空间中分别进行相似度检索，召回 Top-K：
   - 最相关的表/字段结构（Schema）；
   - 相关业务规则与术语定义（Biz Knowledge）；
   - 相似的历史问答对（QA Examples）；
   - 可能相关的样例数据（Sample Rows）；
3. **结果整理与裁剪**：对检索结果按相关度排序、去重与裁剪，控制整体上下文长度，组装为结构化片段；
4. **提供给 Prompt 编排器**：以“【数据库表结构】/【业务知识】/【参考示例】/【样例数据】”等分段形式注入 Prompt。

**同步与更新机制**：
- **Schema 同步**：支持定时全量同步（如每日凌晨）与手动触发增量更新，保证与真实数据库结构一致；
- **业务知识维护**：通过配置文件或管理后台维护业务定义、统计口径与问答对，支持版本控制；
- **索引更新**：当 Schema 或业务知识更新时，自动或按需重建对应命名空间索引，避免全库重建。

### 3.2 Prompt 编排器与 SQL 生成

**角色定位**：Prompt 取代传统 NER/意图识别/句法分析等一系列显式模块，成为驱动 LLM 对问题进行“分步骤推理”的核心“算法”，其质量直接决定 SQL 的正确性和可解释性。

**Prompt 总体结构设计**（结合英语版提示词工程文档）：
1. **System Prompt（系统角色与全局原则）**  
   - 明确模型身份及目标：  
     “你是一个资深数据分析 SQL 助手，需要根据给定的数据库 Schema、业务知识和示例，为用户问题生成高质量的 {dialect} SQL。”  
   - 全局约束与指导原则，例如：
     - 优先使用 JOIN 而非不必要的子查询，注意性能与索引利用；
     - 正确处理时间范围（含起止是否为闭区间）、空值（NULL）等边界情况；
     - 根据问题意图选择合适的聚合、分组和排序方式；
     - 在需要时可以输出简要解释 SQL 设计思路（可配置开启/关闭）。

2. **上下文信息分段（来自 RAG）**  
   按照固定模板组织检索结果，便于模型理解：
   - 【Database schema】  
     - 列出本次检索到的相关表结构、字段、主外键等（DDL/注释），格式化为列表或伪 DDL。
   - 【Business knowledge】  
     - 列出与当前问题相关的业务定义、统计口径、术语解释等。
   - 【Sample records】（可选）  
     - 提供少量示例数据行，帮助模型理解字段语义和取值范围。
   - 【Reference examples】（Few-shot 问答对，可由 RAG 检索或静态配置）  
     - 展示类似问题及其对应 SQL 与简要 rationale（为何这么写）。

3. **User Prompt（用户问题与任务说明）**  
   - 【User question】  
     - 注入用户原始问题；
   - 【Task & Output requirements】  
     - 明确要求：只基于提供的 Schema/业务知识生成 SQL，不虚构表/字段；
     - 指定输出格式，例如：
       - 仅输出 SQL 语句；
       - 或输出 JSON 对象，包含 `sql` 与 `explanation` 字段；
       - 不要输出多条互斥 SQL，如需要多步查询需按约定格式返回。

4. **复杂度与鲁棒性指引**  
   - 针对复杂多表场景，指引模型先“规划再实现”：  
     - 先在思考中列出查询目标、所需表与连接条件、过滤条件、聚合逻辑；
     - 再据此生成最终 SQL（chain-of-thought 只在内部，不返回给用户）。

通过上述模板化结构，Prompt 编排器可以将 RAG 检索结果、业务知识和 Few-shot 示例有机融合，显著提升跨表查询、多维统计等复杂 NL2SQL 任务的成功率。

**Python 实现建议**：
- 定义 `PromptBuilder` 类：
  - `build(user_query, schema_hits, history_summary=None) -> str`
  - 内部模板化拼接系统设定、Schema 信息、对话历史摘要和示例，生成最终 Prompt 字符串。
- 定义统一的 `LLMClient` 抽象层，用于屏蔽不同大模型提供方与部署方式的差异：
  - 核心接口示例：`generate(prompt: str, model: str, **kwargs) -> str`
  - 支持以下多种后端：
    - **国内主流云端大模型服务**：如通义、百川、智谱、腾讯混元等，通过各自官方 API/SDK 调用；
    - **vLLM 等自建推理服务**：通过统一的 HTTP/gRPC 接口调用本地或私有化部署的大模型；
    - 其他符合 OpenAI/兼容协议的通用 LLM 服务。
  - 在此抽象层中统一处理：鉴权配置（API Key、Endpoint）、重试与熔断策略、超时控制、日志与监控埋点，外部调用方只需指定模型标识与参数。

### 3.3 SQL 校验、自我修正与执行

**第一层：语法与安全校验**
- 使用 SQL 解析库（如 `sqlparse` 或特定数据库方言解析器）做初步检查；
- 实施安全策略：
  - 首期 **只允许 SELECT 语句**，拒绝 UPDATE/DELETE/INSERT/DDL；
  - 对敏感表/字段建立黑名单或只读视图；
  - 自动为缺失 `LIMIT` 的查询添加行数上限（如 `LIMIT 1000`）。

**第二层：执行与自我修正闭环**
1. 对通过校验的 SQL，发送到实际数据库执行；
2. 如遇语法错误、字段不存在、权限不足等，捕获详细错误信息；
3. 将错误信息（及原始 SQL、Schema 片段）回注入 LLM，新建 Prompt 请求其**基于错误进行修正**；
4. 控制最大修正轮数（如 2–3 次），避免死循环。

**第三层：结果后处理与解释**
- 将执行结果封装为统一结构（表格数据 + 元信息）返回前端；
- 可选：追加一步 LLM 调用，对查询结果生成**自然语言摘要**或**业务洞察说明**（如“TOP5 产品贡献了 80% 销售额”）。

---

### 3.4 会话与上下文管理

**目标**：支持多轮对话式 NL2SQL，能够理解“接着刚才的查询”“只看华东地区”等省略说法。

**设计要点**：
- 会话存储：
  - 为每个会话维护最近若干轮的（问题、生成 SQL、结果摘要），可存入 Redis 或数据库；
- 上下文摘要：
  - 对较长历史使用 LLM 或规则进行压缩，生成“对话历史摘要”；
  - 在 Prompt 中附带该摘要，帮助 LLM 理解当前问题与历史查询的关联。

初期可以简化为仅使用最近一轮的查询作为上下文，通过实践验证后逐步增强。

## 四、技术栈与基础设施选型

### 4.1 后端与服务框架

- **编程语言**：Python 3.10+
- **Web 框架**：`FastAPI`
  - 提供 RESTful API，自动生成 OpenAPI 文档；
  - 便于与前端/其他服务集成。
- **数据访问**：`SQLAlchemy` 或对应数据库官方驱动。
- **配置与依赖管理**：`poetry` 或 `pip + requirements.txt`。

### 4.2 大模型与向量检索

- **LLM**：通过前述统一 `LLMClient` 抽象接入，可根据部署条件灵活选择或组合：
  - 云端：国内外主流大模型服务（如 Qwen 系列、腾讯混元、通义、百川、智谱、GPT-4/4.1、DeepSeek 等）；
  - 私有化：基于 vLLM 等框架自建的大模型推理服务，或各厂商提供的企业版私有化部署实例。
- **嵌入模型**：
  - 优先选用中文表现较好的模型，如 `qwen3-embedding` / `bge-large-zh` 等。
- **向量检索与存储**：
  - PoC：FAISS（本地文件存储）；
  - 生产：Milvus / pgvector / Elasticsearch dense vector。

### 4.3 运维与监控

- 日志记录：`loguru` 或标准 logging，记录每次请求的：
  - 用户问题、构造的 Prompt、生成的 SQL、执行结果/错误信息、耗时等；
- 监控告警：
  - 接入 Prometheus + Grafana 等，对请求量、错误率、响应时间、LLM 调用失败率等指标进行监控。

## 五、可靠性与安全设计

1. **权限与数据安全**
   - 通过网关或接入层控制用户身份与权限，仅暴露必要的数据范围；
   - 使用视图或数据脱敏策略，避免敏感字段直接暴露给 LLM 或最终用户。

2. **SQL 安全控制**
   - 严格限制生成 SQL 的类型（首期只读查询），禁止高危操作；
   - 对单次查询的扫描数据量、执行时间做限制（如超时自动中断）。

3. **可观测与可追溯**
   - 为每次查询分配唯一 Trace ID，贯穿前后端和 LLM 调用；
   - 保留审计日志，支持问题回溯与分析。

4. **效果评估与持续优化**
   - 基于公开数据集（Spider/WikiSQL）及企业自有问答数据构建离线评测集；
   - 定期统计：
     - SQL 执行成功率（Syntax/Runtime）；
     - 结果正确率（与标准 SQL 结果对比）；
   - 通过 Prompt 优化、RAG 策略调整、少量微调（LoRA 等）迭代提升表现。

## 六、版本规划与演进路线

1. **V1：最小可用版本（MVP）**
   - 只读查询 + 单轮问答；
   - 支持基础 Schema RAG 检索与 SQL 生成；
   - 具备基本日志与错误处理能力。

2. **V2：增强版**
   - 支持多轮对话与上下文理解；
   - 引入自我修正闭环，提高 SQL 执行成功率；
   - 完善监控与安全策略。

3. **V3：面向特定业务域优化**
   - 基于企业历史 `(问题, SQL)` 对 LLM 进行领域微调或 LoRA 适配；
   - 增强对复杂多表 JOIN、嵌套查询、窗口函数等高级特性的支持；
   - 与 BI 平台深度集成，支持可视化图表和自动洞察说明。

## 七、小结

本概要设计以大语言模型为“智能大脑”，通过 RAG、Prompt 编排、SQL 校验与自我修正、Schema 管理和安全控制等模块，构建了一套可在企业场景中实际落地的 NL2SQL 系统方案。后续可以在此文档基础上，进一步细化为详细设计（类图、接口定义、目录结构），并基于统一的 FastAPI 接入层和 LLM 抽象层实现完整的 Python 服务。
