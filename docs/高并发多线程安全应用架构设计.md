# 高并发多线程安全应用架构设计

> 基于《多线程方案分析_ym_new项目.md》中结论与优化方案，面向**大小模型推理框架**的架构级设计。  
> 目标：**高可用、线程安全、支持高并发**。小模型（如 YOLO）采用「视频解码→任务队列→推理」流式管道；大模型（如 LLM/多模态）采用「直接输入文字/图片→推理」请求式管道；二者在架构上**区分设计**，本节先给出区分结论与两套子方案概览。

---

## 1. 设计目标与原则

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| **线程安全** | 多请求、多工作线程并发下无数据竞争与未定义行为，共享状态访问均有明确同步。 |
| **高并发** | 小模型：多通道并行、同通道内解码与推理流水线并行；大模型：多请求并行、任务/工作池调度。 |
| **高可用** | 异常可隔离、可恢复，停止与资源释放可预期、无泄漏。 |

### 1.2 共通设计原则

- **共享可变状态必加锁**：跨线程读写的对象均在统一锁下访问，或通过不可变快照传递。
- **不依赖不可靠原语**：在存在生产-消费队列的场景下，**禁止**用 `Queue.empty()` / `qsize()` 做业务或收尾判断；收尾用「事件 + 带超时的 get」。
- **停止信号可预期**：有长生命周期工作线程时，停止采用 `threading.Event`，由控制方 `set()`，工作线程轮询 `is_set()` 或 `wait(timeout)`。

## 2. 大小模型方案区分分析

### 2.1 是否需要分别设计

**结论：需要针对小模型与大模型分别设计子方案。**

| 维度 | 小模型（如 YOLO） | 大模型（如 LLM / 多模态） |
|------|-------------------|---------------------------|
| **典型流程** | 视频解码 → 任务队列 → 推理检测 | 直接输入文字或图片 → 推理 |
| **输入形态** | 长生命周期流（多路视频/摄像头） | 离散请求（单次文本/图像） |
| **是否需要通道管理** | **需要** | **不需要** |
| **性能关注点** | 按通道隔离、不重复解码、多通道并行、SPSC 流水线 | 请求级并发、批处理（batching）、模型与工作池复用 |
| **线程安全关注点** | 同通道的 start/stop/update 与解码/推理的互斥；每通道 ROI/配置等共享状态 | 模型权重的多线程只读；请求/会话状态；任务队列与工作池的并发访问 |

- **小模型**：一路视频流对应一个「通道」，存在 start/stop/update、持续解码、持续入队、持续推理；若不按通道管理，会重复启动、资源泄漏、同通道控制与状态竞态。因此必须引入**通道管理器**与**按通道的锁**。
- **大模型**：无「流」概念，每次请求自包含；并发体现在多请求同时进入、任务队列与工作池调度。不需要「通道」这一层抽象，只需**请求/任务调度 + 工作池 + 模型与缓存的线程安全访问**。

### 2.2 方案划分

- **子方案 A（小模型）**：视频解码 → 任务队列 → 推理检测；**含通道管理**、SPSC 有界队列、按通道锁与 stop_event、队列收尾不依赖 empty()。见第 3 节及第 4～7 节。
- **子方案 B（大模型）**：直接输入 → 推理；**无通道管理**；请求/任务队列 + 工作池、模型并发只读与可选批处理。见第 8 节。
- **共通部分**：实现清单分列（小模型 / 大模型）。见第 9 节。

## 3. 小模型子架构（视频解码 → 任务队列 → 推理检测）

### 3.1 拓扑与通道管理

```
                    ┌─────────────────────────────────────────────────────────┐
                    │                     API / 请求层                          │
                    │  start(channelId) / stop(channelId) / update(channelId)   │
                    └───────────────────────────┬───────────────────────────────┘
                                                │ 受「按通道全局锁」串行化
                    ┌───────────────────────────▼───────────────────────────────┐
                    │               Channel Manager（通道管理器）                 │
                    │  全局: _channel_objects, _channel_locks, _objects_lock     │
                    └───┬───────────────────────────────────────────────────┬───┘
                        │                                                   │
        ┌───────────────▼───────────────┐                   ┌───────────────▼───────────────┐
        │  Channel A                     │                   │  Channel B                     │
        │  ┌─────────────┐               │                   │  ┌─────────────┐               │
        │  │  Decoder    │── put ───────►│  BoundedQueue     │  │  Decoder    │── put ───────►│  BoundedQueue
        │  │  Thread     │               │  (frame/__stop__) │  │  Thread     │               │  (frame/__stop__)
        │  └─────────────┘               │       │           │  └─────────────┘               │       │
        │         ▲                      │       ▼ get      │         ▲                      │       ▼ get
        │  stop_event                    │  ┌─────────────┐ │  stop_event                    │  ┌─────────────┐
        │         │                      │  │  Inference  │ │         │                      │  │  Inference  │
        │         │                      │  │  Thread     │ │         │                      │  │  Thread     │
        │         └──────────────────────│  └─────────────┘ │         └──────────────────────│  └─────────────┘
        │                                │         ▲       │                                │         ▲
        │                                │  ctx_lock/      │                                │  ctx_lock/
        │                                │  Context        │                                │  Context
        └────────────────────────────────┴─────────────────┴────────────────────────────────┴─────────────────
```

- **请求层**：对同一 `channelId` 的 start/stop/update 在**该通道的全局锁**下串行执行。
- **通道管理器**：维护 `channelId → (context, decoder_future, inference_future, stop_event, …)`，所有对映射的访问在**对象级全局锁**下进行；按 channelId 的锁在首次创建通道时在全局锁内创建并登记。
- **单通道**：解码线程只写队列；推理线程只读队列；二者通过**同一 stop_event** 响应停止；对通道上下文（如 ROI、points）的读写在**该通道的 ctx_lock** 下进行。

### 3.2 核心组件

| 组件 | 职责 | 线程安全要点 |
|------|------|--------------|
| **Channel Manager** | 通道注册/注销、按 channelId 取锁与上下文、生命周期与 Future 管理 | 全局 `_objects_lock` 保护映射；每通道一把 `channel_lock`，与上下文同生命周期 |
| **Bounded Message Queue** | 帧与控制消息（如 `__stop__`）的 SPSC 传递 | 仅用 `put`/`get(timeout)`，**禁止依赖 `empty()` 做逻辑** |
| **Decoder** | 拉流、解码、入队、重连与异常上报 | 停止仅认 `stop_event`；资源仅在解码线程内 release |
| **Inference Worker** | 从队列取帧、执行模型推理、写结果 | 停止仅认 `stop_event`；读通道上下文时持 `ctx_lock` 或读不可变快照 |
| **Stop & Drain** | 统一停止与队列收尾 | 先 `stop_event.set()`，再基于 `get(timeout)` 的循环直到收到 `__stop__` 或超时上限，再注销通道 |

- **小模型特有原则**：按通道隔离、单通道 SPSC、全局可见的按通道锁、停止与收尾使用 Event + 带超时 get。

---

### 3.3 小模型：线程安全与锁设计

1. **全局对象锁 `_objects_lock`**（如 `threading.Lock()`）
   - 保护：`channelId → ChannelContext` 的映射的增删查。
   - 使用范围：仅在「获取/创建/移除通道上下文」的短临界区使用，避免在持锁时做 I/O 或重计算。

2. **按通道锁 `channel_lock`**（每通道一把，存在 `ChannelContext` 内）
   - 保护：该通道的 start/stop/update 语义互斥，以及对该通道上下文中可变字段（如 ROI、points、is_moving）的读写。
   - 使用范围：API 层 start/stop/update；推理线程在需要读 ROI/points 时持同一把锁（或在该锁下生成只读快照供推理线程使用）。

3. **不在锁内做的操作**
   - 解码/推理循环内的长时间计算、网络 I/O、`queue.get(timeout=...)` 等，均在**释放 channel_lock 之后**进行，避免阻塞其他通道或请求。

### 3.4 通道上下文（ChannelContext）设计要点

- **建议字段**（示例）：`channel_id`, `video_or_config`, `message_queue`, `stop_event`, `channel_lock`, `decoder_future`, `inference_future`（若用 Executor），以及业务需要的 `points`, `roi_id`, `is_moving` 等。
- **可变字段**：所有会被 API 线程与推理线程共同访问的可变字段（如 `points`, `roi_id`, `is_moving`），**读写在 `channel_lock` 下进行**；或采用「写时复制」：API 在锁内更新后生成不可变快照，推理线程只读快照引用，避免长时间持锁。

### 3.5 禁止与推荐用法

- **禁止**：使用 `queue.empty()` 或 `queue.qsize()` 作为业务逻辑或收尾逻辑的判断依据（多线程下不可靠）。
- **推荐**：所有「清空队列」或「消费到结束」的语义，均通过「设置 stop_event + 循环 `get(timeout=小值)` 直到收到 `__stop__` 或达到超时/次数上限」实现。
- **禁止**：每请求各持有一份「按 channelId 的锁」；必须使用通道管理器提供的、与通道生命周期绑定的**全局唯一**的 per-channel 锁。
- **推荐**：停止信号统一使用 `threading.Event`，由控制线程 `set()`，工作线程在循环中 `is_set()` 或 `wait(timeout)`。

## 4. 小模型：任务队列设计

### 4.1 接口约定

- **入队**：`put(item)` 或 `put_nowait(item)`；控制消息（如结束）使用约定好的 item（如 `{"__stop__": True}`）。
- **出队**：仅使用 `get(timeout=...)`，超时返回或收到 `__stop__` 作为结束依据；**不提供**依赖 `empty()` 的「清空」接口。
- **容量**：有界队列，容量在通道创建时确定（如 50～150），避免无界堆积导致内存与延迟不可控。

### 4.2 清空与收尾语义

- **停止时收尾**：控制方先 `stop_event.set()`，解码线程检测到后向队列放入一条 `__stop__` 并退出；控制方或收尾逻辑循环 `get(timeout=0.01)` 直到取到 `__stop__` 或连续超时达到上限（如 500 次），再执行通道注销与资源释放。
- **ROI 变更等「丢弃未处理帧」**：不依赖 `empty()`；可在通道锁内设置「flush 标记」或让推理线程在下一帧循环中识别「配置版本变更」并丢弃旧帧；或使用有限次 `get_message(timeout=0.01)`（如最多 100 次）做尽力清空，避免无限循环。

## 5. 小模型：停止与生命周期

### 5.1 正常停止顺序

1. API 收到 stop(channelId)。
2. 在**该通道的 channel_lock** 下：若通道不存在则返回；否则取得 `stop_event` 并 `stop_event.set()`。
3. （可选）若使用 Executor：对 decoder / inference 的 `Future` 调用 `cancel()` 或带超时的 `result(timeout=...)`，再 `executor.shutdown(wait=True)` 或仅 wait 两个 Future。
4. 收尾线程或当前线程：循环 `message_queue.get(timeout=0.01)` 直到取到 `__stop__` 或超时次数达上限。
5. 在**全局对象锁**下从映射中移除该 channelId，释放 channel_lock 与上下文引用，完成注销。

### 5.2 异常与资源释放

- 解码/推理线程内部异常：在各自线程内捕获，向队列放入 `__stop__`（若尚未放入），并确保本线程内释放占用的资源（如 `VideoCapture.release()`）；停止事件由外部或自身在异常路径上设置。
- 所有「释放资源」与「从管理器注销」的顺序：先停止信号 → 再队列收尾（不依赖 empty）→ 再注销映射，避免其他线程仍持有旧引用导致误用。

## 6. 小模型：与 concurrent.futures 的集成（可选）

### 6.1 使用方式

- 使用 **ThreadPoolExecutor**（或每通道一个小的 executor）提交解码任务与推理任务：`decoder_future = executor.submit(decoder_run, ctx)`, `inference_future = executor.submit(inference_run, ctx)`。
- 停止时：先 `stop_event.set()`，再视需要 `decoder_future.cancel()` / `inference_future.cancel()` 或 `result(timeout=...)`，最后做队列收尾与通道注销。
- 好处：统一通过 Future 做超时、取消与异常获取，便于监控与测试。

### 6.2 注意点

- 不改变「每通道 SPSC + 有界队列」的拓扑；Executor 仅用于**线程生命周期管理**，不替代队列作为生产者-消费者通信手段。
- 若使用 ProcessPoolExecutor 做重推理（减轻 GIL），需单独设计帧数据传入（序列化或共享内存）与进程池生命周期，本架构文档不展开。

## 7. 小模型：高并发与可扩展性

### 7.1 通道级并发

- 不同 channelId 之间仅共享「全局对象锁」的短临界区，解码与推理均按通道隔离，因此**通道数增加时吞吐可近似线性扩展**（受限于 CPU/GPU/带宽）。
- 同一 channelId 的 start/stop/update 被该通道的 `channel_lock` 串行化，避免重复启动、覆盖或泄漏。

### 7.2 请求层

- API 层可多线程/多进程处理请求；对同一通道的并发请求会在该通道锁上排队，行为确定且无竞态。
- 通道管理器的映射访问均在全局锁内，临界区应保持极短，仅做字典操作与锁的取用。

## 8. 大模型子架构（直接输入 → 推理）

### 8.1 拓扑与无通道设计

大模型（LLM、多模态等）输入为**离散请求**（单次文本或图像），无长生命周期流，因此**不引入通道管理**。并发由「多请求 + 任务队列 + 工作池」承担。

```
                    ┌─────────────────────────────────────────────────────────┐
                    │                     API / 请求层                          │
                    │  request(text) / request(images) / request(multimodal)   │
                    └───────────────────────────┬───────────────────────────────┘
                                                │ 可多线程/多进程
                    ┌───────────────────────────▼───────────────────────────────┐
                    │               Task Queue（有界请求/任务队列）                │
                    │  入队: (request_id, input, options)；出队: 工作线程       │
                    └───────────────────────────┬───────────────────────────────┘
                                                │ get(timeout=...)
                    ┌───────────────────────────▼───────────────────────────────┐
                    │         Worker Pool（ThreadPoolExecutor / 进程池）          │
                    │  每 worker: 取任务 → 加载/复用模型(只读) → 推理 → 写回结果   │
                    └───────────────────────────┬───────────────────────────────┘
                                                │
                    ┌───────────────────────────▼───────────────────────────────┐
                    │  模型与缓存：多线程只读权重；可选 KV-cache/会话状态加锁      │
                    └───────────────────────────────────────────────────────────┘
```

- **请求层**：接收请求后生成任务入队（或直接 submit 到 Executor），不维护「通道」；可多线程/多进程处理接入。
- **任务队列**：有界 `Queue` 或 Executor 内部队列，仅用于「请求/任务」的投递与领取，**不依赖 `empty()` 做逻辑**（若需「优雅关闭」则用 Event + 带超时 get）。
- **工作池**：多个 worker 并发取任务、调用模型推理；模型权重只读共享，写结果或会话状态时需加锁或线程安全结构。

### 8.2 核心组件

| 组件 | 职责 | 线程安全要点 |
|------|------|--------------|
| **Task Queue** | 请求/任务入队与出队 | 使用 `put`/`get(timeout)`；可选有界容量；不做 empty() 依赖 |
| **Worker Pool** | 取任务、执行推理、写回结果 | 与队列的交互在标准库/Executor 保证下线程安全 |
| **Model / 推理引擎** | 加载权重、前向推理 | 多线程**只读**权重；若框架内部有状态（如 CUDA 流），需按文档保证线程安全或每线程独立上下文 |
| **结果/会话状态** | 写回结果、多轮会话缓存等 | 按请求/会话 ID 隔离或加锁；避免跨请求写同一可变对象 |

### 8.3 线程安全要点（大模型）

- **模型权重**：加载一次、多 worker 只读，无需锁；若使用进程池则每进程一份拷贝或共享内存只读。
- **请求级状态**：每个请求或会话持有独立的结果句柄/回调；不在多请求间共享可变结构除非加锁。
- **任务队列**：使用标准库 `queue.Queue` 或 `concurrent.futures` 的提交顺序；关闭时用 Event + 超时 get 收尾，不用 empty()。
- **批处理（batching）**：若将多请求合并为 batch 再推理，batch 的组包与拆包需在锁或单线程内完成，或使用线程安全的 batch 队列。

### 8.4 高并发与可扩展性（大模型）

- 提高并发：增加 worker 数量（线程或进程）、或增加接入层实例做水平扩展。
- 延迟与吞吐权衡：有界任务队列可背压；可选优先级队列或超时丢弃策略。
- 与「小模型」的共通点：共享可变状态加锁、不依赖 empty()（在存在队列且需收尾时）、异常隔离与资源释放顺序清晰。

## 9. 实现清单（与《多线程方案分析》对应）

### 9.1 小模型（视频解码 → 任务队列 → 推理检测）

| 序号 | 模块/点 | 实现要点 |
|------|---------|----------|
| 1 | Channel Manager | 引入 `_objects_lock`；`save`/`get`/`remove` 通道对象时均在锁内；按 channelId 的锁在创建通道时在全局锁内创建并放入上下文，供所有请求与工作线程使用。 |
| 2 | 通道锁与 API | start/stop/update 入口先取「该 channelId 的 ChannelContext 及 channel_lock」，在 `channel_lock` 下执行逻辑；若通道不存在（stop/update）或已存在（start 重复）则按业务返回。 |
| 3 | 停止与收尾 | 解码/推理线程仅认 `threading.Event`；stop 流程：先 `stop_event.set()`，再循环 `get_message(timeout=0.01)` 直到收到 `__stop__` 或超时次数上限，**不使用 `while not queue.empty()`**；最后再在全局锁内 remove 通道。 |
| 4 | 策略/推理层 | ROI 变更等需「丢弃缓冲」时，改为基于 `get_message(timeout=...)` + 明确条件（条数或 `__stop__`），不依赖 `messageQueue.empty()`；对 `points`/`roi_id`/`is_moving` 的读写在通道的 `ctx_lock`（或与 channel_lock 统一）下进行，或读不可变快照。 |
| 5 | 可选 | 使用 ThreadPoolExecutor + Future 管理解码/推理任务，停止时通过 Event + Future 做统一收尾与超时控制。 |

### 9.2 大模型（直接输入 → 推理）

| 序号 | 模块/点 | 实现要点 |
|------|---------|----------|
| 1 | 任务队列与工作池 | 使用有界 `Queue` 或 `ThreadPoolExecutor`/`ProcessPoolExecutor` 接收请求；仅用 `put`/`get(timeout)` 或 `submit`，不做 empty() 依赖；优雅关闭时用 Event + 超时 get 或 `shutdown(wait=True)`。 |
| 2 | 模型与权重 | 多 worker 仅只读访问模型权重；若多进程则每进程一份或只读共享；框架内部状态（如 CUDA）按文档保证线程安全或每线程独立。 |
| 3 | 结果与会话状态 | 按 request_id/session_id 隔离；跨请求共享的可变结构加锁或使用线程安全容器。 |
| 4 | 批处理（可选） | batch 组包/拆包在单线程或锁内完成，或使用线程安全的 batch 队列，避免竞态。 |
