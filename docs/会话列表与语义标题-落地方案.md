# 会话列表与语义标题 — 落地方案（智能客服 / 算法侧）

> **实现状态（方案 B）**：已在代码中落地 `GET /chatbot/sessions`、Redis `conv:index:{user_id}` / `conv:meta:{user_id}:{session_id}`、内存目录对齐；`app/api/chatbot.py` 内 **ChatbotService 与列表/删除接口共用同一 `ConversationManager`**，避免内存模式下读写分叉。标题一期为 **truncate/off**（`CHATBOT_SESSION_TITLE_*`）。异步 **LLM 标题**仍为后续迭代。

## 0. 文档定位

- **背景**：产品侧需要类似「千问」左侧栏的**会话列表**（按用户维度展示多条会话、可点进查看历史），且标题期望为**语义化短标题**，而非简单截取首句前若干字符。
- **现状**（截至本方案编写时）：
- 已有：`GET/DELETE /chatbot/sessions/messages?user_id=&session_id=`，仅在**已知 `session_id`** 时读写消息；查询返回中 `messages[].content` 为纯文本，图片链接独立在 `messages[].image_urls`。
  - 存储：`conversation/store.py` 以 `(user_id, session_id)` 为键保存消息列表（内存或 Redis `conv:{user_id}:{session_id}`），**无「某用户下全部 session_id」索引**，消息体无 `title` 字段。
  - 鉴权：业务路由使用 **Service API Key**（`Authorization: Bearer`）；**`user_id` 由调用方传入**，算法服务不维护登录用户表。
- **本文**：给出**架构选型、数据模型、接口契约、标题策略、实施阶段与风险**，供评审与排期；**不替代代码**，实现时以本仓库后续 PR 为准。

## 1. 目标与非目标

### 1.1 目标

| 编号 | 目标 | 说明 |
|------|------|------|
| G1 | **按用户列举会话** | 给定 `user_id`，返回该用户在算法侧可见的会话条目（至少含 `session_id`、展示用标题、最近活跃时间等）。 |
| G2 | **标题可演进** | 一期可用**规则/截断**快速上线；二期支持**模型生成语义标题**，体验对齐头部产品。 |
| G3 | **与现有消息存储一致** | 列表与 `GET .../sessions/messages`、对话 `/chat/stream` 使用同一套会话键空间，避免「列表有、点进去没消息」的不一致。 |
| G4 | **可运维** | 与会话 TTL（`CONV_SESSION_TTL_MINUTES`）、单会话消息上限等策略可协调或文档化边界行为。 |

### 1.2 非目标（本方案不强制包含）

- 不实现用户登录、OAuth、JWT；**用户身份仍由调用方认定**，算法侧只信任请求中的 `user_id` 字符串（在已有 Service API Key 前提下）。
- 不实现「对话分组」「置顶」「跨端同步」等纯产品功能的全套后端（可在调用方业务库实现，见第 3 节）。
- 不强制改造 NL2SQL / Analysis 等其它业务的会话列表（可复用同一套 **Session Index** 抽象，但可分期）。

## 2. 架构选型：会话目录放哪？

### 2.1 方案 A — 调用方业务库为「会话目录」主存储（推荐作为默认集成方式）

**思路**：调用方（已登录后台）在自有 DB 维护 `sessions` 表：`user_id`、`session_id`（UUID）、`title`、`updated_at`、分组等；算法服务继续只存 **消息正文**（现有 Redis/内存）。

**优点**

- 与登录态、权限、分组、多端一致性强。
- 算法服务无状态扩展简单，不增加 Redis 扫描与元数据一致性负担。

**缺点**

- 若调用方**未**在首条消息前写入目录，则算法侧已产生 `conv:...` 但列表缺失；需约定「创建会话」时机（如打开新对话页即插一行）。

**适用**：已有用户中心 / BFF 的团队，**优先推荐**。

### 2.2 方案 B — 算法侧 Redis 维护「会话索引 + 元数据」（推荐作为本仓库可选实现）

**思路**：在现有 `conv:{user_id}:{session_id}` 之外，增加索引结构，例如：

- `conv:index:{user_id}` → Redis **Sorted Set**：member = `session_id`，score = `last_activity_ts`（毫秒或秒）；
- 可选 Hash：`conv:meta:{user_id}:{session_id}` → `title`、`title_status`、`created_at`、`updated_at` 等。

**优点**

- 调用方零改造即可拿到「算法侧真实存在」的会话列表。
- 列表排序、分页可与 TTL 策略对齐（见第 6 节）。

**缺点**

- 需在 **每次 append 消息**（及可选删会话）时**同步更新索引**，实现与测试成本高于方案 A。
- 内存版 `ConversationStore` 需维护平行索引结构，或明确「仅 Redis 部署支持列表接口」。

**适用**：希望 **OpenAPI 一站式** 交付、调用方无表或不愿维护会话目录时采用。

### 2.3 方案对比小结

| 维度 | 方案 A（业务库目录） | 方案 B（算法侧索引） |
|------|----------------------|----------------------|
| 一致性责任 | 调用方 + 算法消息 | 算法侧统一 |
| 实现位置 | 调用方 DB + 可选回调 | `store.py` / `manager.py` + 新 API |
| 标题存储 | 业务表字段 | `conv:meta:*` 或 Hash 字段 |
| 推荐 | 有 BFF/用户系统的团队 | 算法服务独立交付、调用方极简 |

**本方案后续接口与数据结构设计以方案 B 为主展开**（便于在本仓库落地）；若选方案 A，可将「元数据字段」映射到业务表，算法侧仅提供 **标题生成建议接口**（可选，见第 5.4 节）。

## 3. 数据模型（方案 B：Redis）

### 3.1 已有（保持不变语义）

- **消息列表**（List）：`conv:{user_id}:{session_id}`，元素为 JSON：`{role, content, ts}`（智能客服带图时 `content` 持久化可附带 `[image_urls]` 块，查询层会拆分为独立 `image_urls` 字段返回）。

### 3.2 新增：会话索引（Sorted Set）

- **Key**：`conv:index:{user_id}`
- **Member**：`session_id`（字符串，与请求体一致）
- **Score**：`last_activity_at`（建议 **毫秒时间戳**，避免同秒多会话排序不稳定；或秒级 + 二级排序规则文档化）
- **操作**：
  - `append_message` 成功后：`ZADD conv:index:{user_id} last_activity_at session_id`（已存在则更新 score）
  - `clear(user_id, session_id)`：`ZREM conv:index:{user_id} session_id`，并删除 `conv:meta:...`（若存在）

### 3.3 新增：会话元数据（Hash，可选但推荐）

- **Key**：`conv:meta:{user_id}:{session_id}`
- **建议字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `title` | string | 展示标题（截断或模型生成结果） |
| `title_source` | string | `truncated` / `llm` / `user`（预留用户重命名） |
| `title_updated_at` | float | 标题最后更新时间 |
| `first_user_preview` | string | 首条 user 前 N 字（缓存，减少读整条 List） |
| `message_count` | int | 可选，便于展示；可与 List 长度定期对齐或每次 append 自增 |

**说明**：`title` 可在首条 user 写入后异步更新；列表接口优先读 Hash，缺失时回退 `first_user_preview` 或现场截断。

### 3.4 内存版 ConversationStore

二选一（需在实现时写清）：

- **M1**：内存结构增加 `Dict[user_id, SortedDict/heap]` 或 `Dict[user_id, List[SessionSummary]]`，行为对齐 Redis。
- **M2**：列表接口在内存模式下返回 **501 或 503**，文档说明「生产请使用 `REDIS_URL`」。

建议 **M1** 便于本地联调，规模注意仅适用于 dev。

## 4. API 设计

### 4.1 列举会话

- **方法 / 路径**：`GET /chatbot/sessions`
- **鉴权**：与现有业务路由一致，`Authorization: Bearer <SERVICE_API_KEY>`。
- **Query 参数**（建议）：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `user_id` | string | 是 | 与现有接口一致，调用方用户标识 |
| `limit` | int | 否 | 默认 50，上限 200（可配置 `CONV_SESSION_LIST_MAX`） |
| `cursor` | string | 否 | 游标分页（见下）；首屏不传 |
| `order` | string | 否 | `desc`（默认，最近活跃在前） / `asc` |

- **游标分页（建议）**：使用 `(last_activity_at, session_id)` 元组做 **composite cursor**（Base64url 编码），避免纯 offset 在大 ZSET 上不稳定。若一期求简，可用 `offset` + `limit`，文档注明生产改用游标。

- **响应体（示例 JSON 形状）**：

```json
{
  "user_id": "u1",
  "items": [
    {
      "session_id": "s-uuid-1",
      "title": "HTTPX与Requests库的区别",
      "title_source": "llm",
      "last_activity_at": 1710000000000,
      "message_count": 12
    }
  ],
  "next_cursor": null
}
```

### 4.2 与现有接口关系

| 接口 | 作用 |
|------|------|
| `GET /chatbot/sessions` | 列表（本方案新增） |
| `GET /chatbot/sessions/messages` | 某会话全文/分页消息（已有） |
| `DELETE /chatbot/sessions/messages` | 删除会话消息与索引、meta（需在实现中扩展：不仅删 List） |

### 4.3 可选：仅生成标题（供方案 A）

若会话目录在调用方 DB，可提供内部或受控接口：

- `POST /chatbot/sessions/title-suggest`  
  Body：`{ "user_id", "session_id" }` 或 `{ "first_query": "..." }`  
  Response：`{ "title": "..." }`  

需 **限流 + 仅 Service API Key**，避免被滥用刷 LLM。

## 5. 标题生成策略

### 5.1 一期：零模型 — 截断与清洗

- 取**首条 `role=user` 的 `content`**，去首尾空白，按 **Unicode 字符数**截断（如 24～32），加省略号或省略不加（产品定）。
- 写入 `title_source=truncated`，`title_updated_at=now`。

**优点**：无额外延迟与费用。  
**缺点**：长句、口语化问法标题质量一般。

### 5.2 二期：模型生成语义标题（对齐千问类体验）

- **触发时机**（推荐）：首条 user 消息成功 `append` 后，**异步任务**调用小模型或主模型短提示生成标题（例如限制 20 字以内、禁止 Markdown）。
- **Prompt 要点**：仅输入用户首问（或首问 + 可选意图标签），输出单行标题。
- **写入**：成功后更新 `conv:meta:*` 的 `title`、`title_source=llm`、`title_updated_at`；失败则保留截断标题。
- **并发**：同一会话加分布式锁或「仅当 `title_source==truncated` 时允许 LLM 覆盖」，避免重复调用。

### 5.3 配置项（建议）

| 环境变量 | 含义 | 默认建议 |
|----------|------|----------|
| `CHATBOT_SESSION_TITLE_MODE` | `off` / `truncate` / `llm` | `truncate` |
| `CHATBOT_SESSION_TITLE_MAX_RUNES` | 截断模式最大字符数 | `28` |
| `CHATBOT_SESSION_TITLE_MODEL` | 可选，专用小模型名 | 空则复用默认 LLM |

### 5.4 流式对话下的体验

- 用户发送首条后列表可先显示 **截断标题**，数秒后 WS/SSE 推前端刷新，或前端轮询 `GET /chatbot/sessions`（节流）。

## 6. TTL、删除与一致性

### 6.1 与 `CONV_SESSION_TTL_MINUTES` 的关系

- Redis 上对 `conv:{user_id}:{session_id}` 已有 **EXPIRE**（若配置 TTL）。  
- **索引与 meta** 必须与主键 **相同 TTL**，或在 key 过期后用 **惰性清理**（列表查询时发现主 key 不存在则 `ZREM` + 删 meta）。  
- 推荐：**每次写消息时同时对 index / meta / message list 续期或设置相同 TTL**，保证一致过期。

### 6.2 `DELETE /chatbot/sessions/messages`

- 除删除消息 List 外，必须 **`ZREM` + `DEL meta`**，否则列表出现「幽灵会话」。

### 6.3 多 worker

- 方案 B 全在 Redis 时与现有多实例一致；内存模式不推荐生产列表。

## 7. 安全与滥用

- **Service API Key** 不变；`user_id` 仍由调用方传。需认知：持有 API Key 的调用方可枚举任意 `user_id` 的会话列表（若不做额外约束）。
- **可选加固**（按需排期）：
  - 调用方 BFF 只转发「已登录用户自己的 `user_id`」；
  - 或算法侧增加 **HMAC 签名的 user_token**（超出本方案范围）。

## 9. 测试与验收

- **单测**：append 后 index score 更新；clear 后列表无该项；TTL 过期后列表行为符合文档。  
- **集成**：先 `POST /chat/stream` 创建会话，再 `GET /chatbot/sessions`，再 `GET .../messages` 内容一致。  
- **性能**：大 `user_id` 下 ZSET `ZREVRANGE` + `LIMIT`；避免 `KEYS`，使用 `SCAN` 仅用于运维。

## 12. 实现后复检（已知边界）

- **跨业务复用键**：Chatbot、NL2SQL、Analysis、LLM 推理各自 `ConversationManager()` 时，在 **Redis 下共用** `conv:{user_id}:{session_id}`；相同二元组会共享一条消息列表，需在集成侧区分 `session_id`。
- **Redis 写入时序**：`append_message` 异步提交，同一请求内若立即依赖「已持久化」再读，存在极小竞态；流式对话正常在生成结束后再读一般无感。
- **`user_id` 含冒号**：当前 key 为字符串拼接，若 ID 含 `:`，可能与解析逻辑或运维脚本冲突，建议使用 URL 安全子集（字母数字、`-`、`_`）。
- **升级前 Redis 数据**：仅存在 `conv:{uid}:{sid}` 列表、无 ZSET 索引的旧会话，在首次新消息写入前 **不会** 出现在 `GET /chatbot/sessions`（与设计一致）。
- **列表接口故障**：Redis 版 `list_sessions` 失败时返回空列表并打错误日志，**不再**回退到进程内空字典，以免掩盖故障。
- **企业级加固（实现）**：`get_default_store()` 进程内单例；`user_id`/`session_id` 经 `app/conversation/ids` 校验（默认禁止 `:`）；`ConversationIdValidationError` → HTTP 422；Redis `get_recent_history` / `get_messages` / `clear` 失败不回退假内存视图；可选 `CONV_REDIS_APPEND_WAIT_MS`；Redis 连接/读超时与健康检查参数见 `.env.example`。
