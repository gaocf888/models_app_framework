# 智能客服上下文理解增强优化方案（2026-04-30）

## 1. 背景与问题

当前智能客服已具备：

- 会话热存储（Redis / 内存回退）；
- 会话冷存储（EasySearch 归档）；
- 多轮上下文拼接（`enable_context=true` 时读取历史）；
- 既有意图识别、RAG、流式输出链路。

在线上使用中，针对“请详细分析一下上述第7点”这类**结构化指代问题**，存在理解偏差。核心原因：

- 当前上下文记忆以“历史原文拼接”为主，缺少“第 N 点”可寻址索引；
- 当启用 RAG 时，外部检索内容可能稀释对会话内部结构引用的注意力；
- 对“上文结构项引用”缺乏专门的前置解析与约束注入。

> 目标：在**不改变现有智能客服主链路和主逻辑**的前提下，新增一层“结构化摘要/索引”能力，提升上文结构引用理解准确率。

---

## 2. 设计原则（约束）

1. **不改主链路**  
   不重构现有 `/chatbot/chat`、`/chatbot/chat/stream`、LangGraph/legacy 路径，不改变既有 RAG、会话写入、流式协议等主流程。

2. **增强/叠加而非替换**  
   对“上文结构引用”的识别与处理，作为现有意图识别能力的增强层，叠加在当前链路上，不替换已有意图分类策略。

3. **异步优先**  
   结构化摘要/索引提取异步化，不阻塞用户回答，不拉长首字节与整体响应时间。

4. **冷热分层**  
   Redis 热存储负责在线低延迟引用解析；EasySearch 冷存储负责归档、追溯、离线分析。

5. **配置复用（关键澄清）**  
   结构化摘要/索引热存储的“轮次保留策略”，复用当前会话热存储相同环境变量（详见第 5 节）。

---

## 3. 总体方案

在现有“原始会话消息热/冷存储”之外，新增一条并行“结构化索引副链路”：

1. assistant 回答完成后（落库后）触发异步任务；
2. 异步任务对回答做结构化提取（要点编号、段落项、关键词）；
3. 写入结构化索引热存储（Redis）；
4. 同步写入结构化索引冷存储（EasySearch）；
5. 下一轮请求命中“上文结构引用”时，优先用热索引解析并注入明确引用上下文。

该副链路不改变现有主问答路径，仅在“命中结构化引用场景”时增强上下文理解。

---

## 4. 存储设计

### 4.1 既有存储（保持不变）

- 热层：会话消息（Redis `conv:*`）；
- 冷层：会话消息归档（EasySearch `conversation_messages_v1`）、会话目录索引（`conversation_sessions_v1`）。

### 4.2 新增结构化索引热存储（Redis）

建议 key（示例）：

- `conv:outline:{user_id}:{session_id}`：会话级结构化索引容器；
- `conv:outline:{user_id}:{session_id}:{assistant_message_id}`：单轮回答结构化条目。

建议字段（示例）：

- `assistant_message_id`
- `turn_ts`
- `items`: `[{idx, title, gist, keywords, raw_span}]`
- `source`: `rule | llm`
- `status`: `success | failed`

### 4.3 新增结构化索引冷存储（EasySearch）

新增索引（建议）：`conversation_outline_v1`

建议字段：

- `user_id`, `session_id`, `assistant_message_id`
- `item_idx`, `item_title`, `item_gist`, `keywords`
- `ts_ms`, `created_at_ms`
- `extractor_version`, `source`

用途：

- 冷归档追溯；
- 离线评估“第 N 点命中率”；
- 热层 miss 时兜底查询（可选，不建议在线高频启用）。

---

## 5. 配置策略（含用户澄清）

### 5.1 轮次保留策略复用现有变量（已澄清）

结构化索引热存储的保留轮次，与当前会话热存储使用同一环境变量策略：

- 复用 `CONV_MAX_HISTORY_MESSAGES` 作为“可索引轮次/条目上限”的主约束；
- 复用 `CONV_SESSION_TTL_MINUTES` 作为热索引过期策略；
- 不新增独立“outline retention”变量（除非后续经压测证明必要）。

这样可保证热消息与热索引在生命周期上对齐，降低配置复杂度与运维负担。

### 5.2 其他建议开关（可新增）

- `CHATBOT_OUTLINE_ENABLED=true`
- `CHATBOT_OUTLINE_ASYNC_ENABLED=true`
- `CHATBOT_OUTLINE_RULE_FIRST=true`
- `CHATBOT_OUTLINE_ES_ENABLED=true`
- `CHATBOT_REFERENCE_RESOLVE_ENABLED=true`

> 说明：以上为新增建议，不影响现有变量语义。

---

## 6. 提取与解析流程

### 6.1 结构化提取（异步）

触发时机：

- assistant 回答完成并写入会话后。

提取策略：

1. **规则优先**（低成本）  
   识别 `1. 2. 3.`、`（一）（二）`、`-` 列表、分段标题等；
2. **LLM 回退**（规则失败时）  
   让模型输出标准 JSON 结构（带 item idx 与 gist）。

### 6.2 上文结构引用解析（在线）

在请求进入最终模型生成前，新增“轻量引用解析步骤”：

1. 检测 query 是否命中结构引用模式（如“上面第7点/上一条第2项/前文第三条”）；
2. 命中则从 Redis 结构化索引读取对应 item；
3. 将解析结果以显式上下文块注入（例如 `resolved_reference`）；
4. 未命中则保持原有链路。

---

## 7. 意图识别增强方式（已澄清）

不替换现有意图识别，仅增强/叠加：

1. 在现有意图识别前后增加一个“结构引用识别器”（规则 + 轻量分类）；
2. 输出额外标记，例如：
   - `has_structured_reference=true`
   - `reference_type=ordinal_item`
   - `reference_index=7`
3. 将标记作为现有意图识别结果的附加特征，不改动原标签集合与主路由语义。

这样可以最大限度保证与当前线上行为兼容。

---

## 8. 与 RAG 的协同策略

当命中“上文结构引用”时，建议优先级：

1. `resolved_reference`（最高）
2. 会话历史原文
3. 外部 RAG 检索片段

可执行策略（渐进）：

- 策略 A：命中结构引用时，RAG 仍开，但降低权重；
- 策略 B：命中结构引用且置信度高时，跳过本轮外部 RAG；
- 策略 C：命中但索引缺失，回落原链路（历史 + RAG）。

---

## 9. 实施计划（不破坏现网）

### 阶段 1：索引生成与存储（旁路）

- 新增结构化提取异步任务；
- 写 Redis 热索引 + EasySearch 冷索引；
- 不改变在线问答输入拼接逻辑。

### 阶段 2：在线引用解析（受控开关）

- 开启 query 结构引用检测；
- 命中后注入 `resolved_reference`；
- 通过开关灰度，不影响全量流量。

### 阶段 3：意图识别增强与 RAG 优先级调优

- 将结构引用信号叠加到现有意图识别结果；
- 对命中引用场景启用 RAG 降权策略；
- 观测命中率与回答准确率后再逐步放量。

---

## 10. 监控与评估指标

建议新增指标：

- `chatbot_outline_extract_total/success/fail`
- `chatbot_reference_resolve_total/hit/miss`
- `chatbot_reference_resolve_latency_ms`
- `chatbot_reference_query_accuracy_sampled`（抽样人工评估）
- `chatbot_rag_suppressed_on_reference_total`

目标观察项：

- “第 N 点”类问题命中率、正确率提升；
- 首字节时间与总响应时延无明显退化；
- 异步失败不影响主流程可用性。

---

## 11. 风险与回滚

风险：

- 结构提取质量不稳定（规则覆盖不足）；
- 解析误命中导致错误引用；
- 新增索引写入带来少量存储与计算开销。

控制：

- 全流程开关化；
- 异步失败自动降级到原链路；
- 命中置信度不足时不注入引用上下文。

回滚：

- 关闭 `CHATBOT_OUTLINE_ENABLED` / `CHATBOT_REFERENCE_RESOLVE_ENABLED`，即回到当前线上行为。

---

## 12. 结论

本方案在保持现有智能客服主链路稳定的前提下，通过“结构化索引副链路 + 上文引用解析增强”，可显著改善“上面第 N 点”类跨轮指代理解问题，且具备低侵入、可灰度、可回滚的工程可实施性。
