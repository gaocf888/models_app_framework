| 能力 | 代码实现状态 | 环境 / 配置状态 |
|------|--------------|------------------|
| Chatbot LangGraph 链路 | **已完成**：`ChatbotService` 已接入 `ChatbotLangGraphRunner`（`fault_case_gate` 相似案例门控 + 意图路由 + C-RAG + Runner 层流式与相似案例追加 + SSE `meta`）；保留 legacy 回退与非流式兼容接口 | 需部署 vLLM（多模态用于故障图检时）；配置 `LLM_DEFAULT_ENDPOINT` 与 `CHATBOT_*`；可选 `CHATBOT_SIMILAR_CASE_*` / `CHATBOT_FAULT_*`；checkpoint-redis 见 `CHATBOT_CHECKPOINT_REDIS_URL` |
| 综合分析 LangGraph 链路 | **已完成企业版 V2**：**`AnalysisImgDiagGraphRunner`** + **`AnalysisService`**，`payload`/`nl2sql` 两套 **`AnalysisGraphRunner`** LangGraph；**`run-img-diag`** 并行编排（视觉 ‖ NL2SQL ‖ 业务 RAG）；节点追踪、RAG 证据与退化治理 | 同上；看图诊断需配置 **`img_diag_*`** 与多模态 endpoint；可继续增强集成测试与可视化编排能力 |

| LangSmith 集成 | **已完成中间层**：`LangSmithTracker` 提供可配置的 `log_run` 能力，已挂在 Chatbot/LLMInference/Analysis/NL2SQL 链路上 | 需在环境中部署 LangSmith，配置 `LANGSMITH_API_KEY`、`LANGSMITH_PROJECT`、`LANGSMITH_ENABLED` 等环境变量 |

### 2. 会话与 RAG 持久化（Redis + 向量库）（P4/P5/P6/P11）

| 能力 | 代码实现状态 | 环境 / 配置状态 |
|------|--------------|------------------|
| 会话 Redis 持久化 | **已完成**：`ConversationStore` 支持内存与 Redis，`ConversationManager` 默认通过 `get_default_store()` 选择实现；支持基础 TTL/摘要策略 | 需部署 Redis 实例，并配置 `REDIS_URL`；可按需要调整会话过期与摘要参数 |
| 通用 RAG & 摄入 | **已完成骨架**：`RAGService` + `RAGIngestionService` + `/rag/ingest/texts` & `/rag/datasets` 支持文本摄入与数据集检索 | 当前默认使用 FAISS 文件版向量库（持久化文件为 `faiss.index` + `faiss_meta.json`，两者随向量数与文本条数线性增长）；大规模场景需关注文件体量、内存占用与启动时间，并按架构文档建议演进为分 shard/分 namespace 或专用向量数据库（如 pgvector/Milvus） |
| GraphRAG & 混合检索（可选） | **已完成骨架**：`GraphIngestionService` + `GraphQueryService` + `HybridRAGService` 以及 `RAGConfig.graph` 配置结构，支持在代码层根据配置选择仅向量 / 仅图 / 混合检索 | 需部署 Neo4j 图数据库并安装大模型应用依赖（`neo4j`、`langchain-community`）；通过环境变量 `GRAPH_RAG_ENABLED`、`NEO4J_URI`、`NEO4J_USERNAME`、`NEO4J_PASSWORD`、`GRAPH_RAG_MODE` 等开启 GraphRAG，当前默认配置仍为纯向量 RAG，实体/关系抽取与图查询策略可按业务需要逐步完善 |
| NL2SQL 专用命名空间 | **已完成骨架**：`NL2SQLRAGService` 定义 `nl2sql_schema` / `nl2sql_biz_knowledge` / `nl2sql_qa_examples` 三类命名空间，并在 NL2SQL 查询时联合检索；`RAGIngestionService` 与 `/rag/ingest/texts` 支持 `namespace` 字段 | 需在实际项目中将 Schema/业务规则/问答样例数据批量写入上述命名空间，并结合真实向量库评估检索效果 |

### 7. 总结与后续建议

- **代码侧**：当前仓库已经完成从架构设计到企业级骨架代码的主要实现，涵盖了大模型/小模型/RAG/会话/NL2SQL/训练管理/监控等所有核心能力，且与 `memory-bank` 文档保持一致。  
- **环境侧**：下一阶段工作重点应转向：
  - 在目标环境中部署并配置必要的基础设施（vLLM、Redis、数据库、向量库、LangSmith、Prometheus/Grafana/Loki 等）；
  - 使用 `benchmarks/` 与 `docs/性能与稳定性评估.md` 完成至少一轮系统性压测与调优；
  - 在真实业务场景中验证 Chatbot、综合分析、NL2SQL、小模型通道的效果，并将经验沉淀回本仓库的文档（尤其是 `大小模型应用技术架构与实现方案.md` 与 `性能与稳定性评估.md`）。
