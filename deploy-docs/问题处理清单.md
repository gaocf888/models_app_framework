# 问题处理清单（部署与运行）

本文档汇总本轮上线中出现的关键问题、已落地处理方案与后续预防建议，便于运维快速排障。

---

## 1. `models-app` Python 解释器与依赖环境错位

### 现象
- 容器内启动使用系统 Python（如 `/usr/bin/python3`），但 CUDA / torch 在 conda 环境；
- 表现为重排器或其他模型调用时设备异常、包版本异常。

### 根因
- 启动命令与依赖安装不在同一 Python 环境；
- 镜像中同时存在系统 Python 与 conda Python，路径优先级不一致。

### 已处理
- `app/app-deploy/docker-mx/Dockerfile-mx`
  - `ENV PATH=/opt/conda/bin:${PATH}`
  - 使用 `/opt/conda/bin/pip` 安装依赖
  - `CMD` 改为 `/opt/conda/bin/python -m uvicorn ...`
- `app/app-deploy/docker-mx/docker-compose-mx.yml`
  - `models-app` / `models-app-gpu` `command` 统一为 `/opt/conda/bin/python -m uvicorn ...`

### 验证
- `ps` 进程应显示 `/opt/conda/bin/python -m uvicorn ...`；
- `python -c "import sys;print(sys.executable)"` 输出 `/opt/conda/bin/python`。

## 2. `sentence-transformers` 导入触发 `torchcodec` 失败

### 现象
- 启动时报错：`Could not load libtorchcodec`、`libxcb.so.1` 缺失；
- EmbeddingService 初始化失败，导致服务起不来。

### 根因
- `sentence-transformers` 5.x 导入链涉及多模态 `torchcodec`；
- 当前定制 PyTorch / 系统依赖与 `torchcodec` 不兼容。

### 已处理
- `requirements-大模型应用.txt`
  - 将 `sentence-transformers` 约束为 `<5.0.0`
- `app/app-deploy/docker-mx/Dockerfile-mx`
  - 增加 `libxcb`（兜底系统依赖）

### 验证
- 容器内 `pip show sentence-transformers` 为 4.x；
- 服务启动不再出现 `libtorchcodec` 导入错误。

## 4. `/chatbot/chat/stream` 带图请求 vLLM 返回 400（URL 非法）

### 现象
- `vllm-service` 报错：`The URL must be either a HTTP, data or file URL.`

### 根因
- 图片预处理后被改写为相对路径（如 `/chatbot/media/xxx.jpg`）；
- vLLM 多模态严格要求绝对 URL（http/https/data/file）。

### 已处理（方案 B：MinIO）
- 目标：预处理后上传 MinIO，向 vLLM 传预签名 URL。
- 代码改动：
  - `app/services/chatbot_image_preprocessor.py`
    - 新增 MinIO 上传与预签名 URL 返回逻辑
    - 存储后端支持 `minio/local`（默认 `minio`）
  - `app/core/config.py`
    - 新增 `CHATBOT_IMAGE_STORAGE_BACKEND` 与 `CHATBOT_IMAGE_MINIO_*` 配置映射
  - `requirements-大模型应用.txt`
    - 新增 `minio` 依赖
- 部署改动：
  - `app/app-deploy/docker-compose.yml`
  - `app/app-deploy/docker-mx/docker-compose-mx.yml`
    - 新增 `minio` 服务、数据卷、网络与依赖
  - `app/app-deploy/.env.example`
    - 新增 MinIO 相关参数

### 验证
- `models-app` 侧日志中图片 URL 应为 MinIO 预签名 `http(s)://...`；
- `vllm-service` 不再出现 URL scheme 400。

## 5. 知识摄入卡在 `mineru_route`，MinerU 无处理日志

### 现象
- 摄入任务 step 显示 `mineru_route`；
- `mineru-api` 无请求日志；
- 常发生在“处理中重启服务”后。

### 高概率根因
1. 进程内异步任务（线程池）在重启时中断，状态遗留为 RUNNING；
2. MinerU Redis 并发门闩 token 泄漏，后续任务卡在 gate 获取，未真正发起 MinerU 请求。

### 已处理（最小修复版）
- `app/rag/ingestion_orchestrator.py`
  - 启动时自动回收“僵尸 RUNNING”任务为 FAILED
  - 新增 `recover_stuck_jobs(...)`
  - 标记错误码：`E_ORPHANED_AFTER_RESTART`
- `app/api/rag_admin.py`
  - 新增接口：`POST /rag/jobs/recover_stuck?threshold_seconds=...`
- `app/rag/mineru_ingest.py`
  - 增加 MinerU gate 等待耗时日志（可判断是否卡在 semaphore）
- `app/core/config.py` + `.env.example`
  - 新增：
    - `RAG_RUNNING_STUCK_TIMEOUT_SECONDS`
  - 启动恢复改为强制启用：重启后会自动回收并重排队遗留 RUNNING 任务（不再提供开关）。

### 当前现场处理步骤（可直接执行）
1. 先确认应用实际使用的 Redis DB（避免在错误 DB 里排查）：
   - `docker exec -it models-app sh -lc 'echo REDIS_URL=$REDIS_URL'`
   - 若为 `redis://.../1`，后续 `redis-cli` 需用 `-n 1`；若 `/0` 则用 `-n 0`。
2. 在正确 DB 检查 semaphore 状态（注意：键名不是命令，必须配合 `GET/LLEN/...` 使用）：
   - `docker exec -it models-app-redis redis-cli -n <DB号>`
   - `GET mineru:ingest:sem_pool_initialized`
   - `LLEN mineru:ingest:sem_pool`
   - `LRANGE mineru:ingest:sem_pool 0 -1`
   - `KEYS mineru:ingest:*`
3. 若判定 gate 泄漏（常见：`initialized=1` 且 `LLEN=0` 且无真实执行任务），重置 semaphore pool：
   - `DEL mineru:ingest:sem_pool`
   - `DEL mineru:ingest:sem_pool_initialized`
   - `DEL mineru:ingest:sem_pool_init_lock`
4. 重启应用容器使 gate 重新初始化：
   - `docker restart models-app`
5. 回收僵尸任务并重试：
   - `POST /rag/jobs/recover_stuck?threshold_seconds=600`
   - `POST /rag/jobs/{job_id}/retry`
6. 双端日志联查确认已真正发起 MinerU：
   - `docker logs -f models-app`
   - `docker logs -f mineru-api`
   - 期望先出现 `MinerU gate acquired ...` / `MinerU file_parse start ...`，随后 `mineru-api` 出现 `/file_parse` 请求日志。

### 后续建议
- 任务系统从进程内线程池逐步升级到外部队列（Celery/RQ 等）；
- MinerU gate 增加租约/过期回收机制，避免 token 永久泄漏。
