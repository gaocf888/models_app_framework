## 1. 从使用视角看整体流程

在当前基座中，小模型训练的代码骨架尚未完整实现为统一的 API/Service，但整体预期流程如下：

1. 通过独立的训练脚本（如 `train_det.py` / `train_cls.py` 等）在训练环境中完成小模型训练；  
2. 将训练得到的权重文件（如 `.pt` / `.onnx` / `.engine`）注册到 `SmallModelRegistry` 中，填写 `weights_path` 与 `task_type` 等元数据；  
3. 由小模型通道推理（见《小模型通道推理整体实现技术说明》）在运行时加载相应权重并执行实时推理；  
4. 训练过程中的日志与指标通过 TensorBoard / Prometheus 等工具进行可视化与监控。

## 2. 模块与文件映射（现状）

目前仓库中对小模型训练的支持主要体现在以下骨架设计与文档说明中：

| 模块 | 路径 | 说明 |
|------|------|------|
| 小模型注册表 | `app/small_models/registry.py` | 维护小模型算法元数据（包括训练产出的权重路径），供推理时使用。 |
| 小模型推理引擎 | `app/small_models/inference_engine.py` | 通过 `SmallModelRegistry` 查找权重路径与 task_type，实际推理逻辑在生产环境中扩展。 |
| 小模型通道管理 | `app/small_models/channel_manager.py` | 不直接负责训练，但与训练产出的模型高度相关。 |
| 架构文档 | `docs/大小模型应用技术架构与实现方案.md`（4.2） | 对“小模型推理检测与训练（代码 + TensorBoard 可视化）”进行总体描述。 |
| 组件清单 | `memory-bank/02-components.md`（小模型相关条目） | 列出 `SmallModelRegistry`、`SmallModelInferenceEngine`、`SmallModelTrainingService` 等组件的职责。 |

> **说明**：`SmallModelTrainingService` 在当前代码中主要以“规划中的组件”形式存在，具体实现可根据后续业务需求和现有训练脚本逐步补充。

## 3. 与小模型通道推理的关系

小模型训练与推理在工程上的关系可以概括为：

- **训练产出 → 注册 → 推理加载**：  
  - 训练脚本产出权重文件与必要的配置（如输入尺寸、类别数等）；  
  - 通过 `SmallModelRegistry` 进行注册，使之对 `model_name` 或 `task_type` 可见；  
  - 通道推理在运行时根据通道配置中的 `model_name` / `algor_type`，从注册表中读取对应元数据并加载权重。  
- **监控与可视化共享**：  
  - 训练过程中的损失/精度等指标通过 TensorBoard 展示；  
  - 推理过程中的 QPS/延迟/错误率等指标通过 Prometheus + Grafana 展示；  
  - 二者共同构成“小模型全生命周期监控”闭环。

详细的小模型通道推理实现，请参考：《小模型通道推理整体实现技术说明》。

## 4. 配置与依赖建议

- 依赖文件：`requirements-小模型训练.txt`  
  - 建议按具体算法栈加入如 PyTorch、TorchVision、MMDetection、MMYOLO、TensorBoard 等依赖；  
  - 若使用 ONNX/TensorRT，则补充 onnxruntime、tensorrt 等包。  
- 配置管理：  
  - 训练脚本可使用 YAML/JSON 配置文件描述模型结构、数据集路径、训练超参等信息；  
  - 训练完成后在一个统一的“小模型注册配置”文件（或数据库表）中登记模型元数据，与 `SmallModelRegistry` 对齐。  
- 与大模型训练的协同：  
  - 小模型训练通常不依赖 RAG/NL2SQL，仅在日志、监控与部署流程上与大模型训练保持一致。  
  - 在一体化平台场景下，可复用“大模型训练微调”中的任务管理思路（如使用类似 Orchestrator 的模式管理小模型训练任务）。
