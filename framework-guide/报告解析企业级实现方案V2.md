# 报告解析企业级实现方案 V2

> **定位**：在 **不依赖** 现有 `DocumentParser` 将 Word 表格压成 `[DOCX_TABLE] + 管道符` 扁平文本作为**唯一真源**的前提下，重新设计「检修报告 → 结构化 JSON」的企业级管线。V2 强调 **Word 专用摄入（python-docx 为主）**、**分块处理**、**程序保表格几何 + 多策略启发式 + LLM 补语义** 的混合架构。  
> **与 V1 关系**：V1（`报告解析企业级实现方案.md`）描述的是以当前 RAG 侧 `DocumentParser` 与多阶段 LLM 为主的落地方案；**V2 为独立演进线**，实现时可 **替代或并存** 检修模块的解析与编排逻辑，**不以 V1 为必须前提**。

---

## 0. 已确认方案点（与产品/实现锁定）

以下结论已确认，实现与评审以本节为准；**若后续变更需同步更新本文与配置说明**。

| 项 | 结论 |
|----|------|
| **与现网关系** | V2 与当前「检修报告结构化提取」**并行运行**，**暂不替换** V1；代码 **完全隔离**（独立包/入口），便于后续只删 V1 而不牵连其它模块。通过环境变量 **特性开关** 选择管线（`INSPECT_EXTRACT_PIPELINE_VERSION`，取值 `v1` 或 `v2`）。 |
| **Phase 1 范围** | **docx 与 pdf 必须同期可用**（同一次发布中两种入口均可走 V2 管线；pdf 仍遵循「文本层 / MinerU」路由思想）。 |
| **业务规则权威** | 管号正负、行号、检测位置在边界场景（如「通用」「2-6」「右墙 1-1」等）的命名与映射，以 **原始 Word 需求与示例表** 为权威；实现时可对照 **需求截图**（示例：典型表格与示例输出对照图）及 **`configs/prompts.yaml`** 中检修相关提示词版本作为辅助说明，**冲突时以需求/示例为准**。 |
| **docx 底纹（Phase 1）** | **必须**支持从 docx 单元格读取 **底纹填充（shading）**，并映射为 **「超标候选」** 特征进入下游（与正文阈值规则并存；避免仅靠正文复述）。 |
| **对外契约** | **响应 JSON 与当前接口完全一致**（字段名、枚举、`records`/`summary`/`trace` 等）；后续若调整契约再单独评审。 |
| **同步接口** | **保留**现有同步 `POST .../run`（与现网一致）。**异步任务、按块拉取、断点续跑** 作为企业级可选能力，目标架构见 **§15**；Phase 1 是否交付异步端点以排期为准，但设计与实现应与之对齐。 |

## 1. 背景与为什么要 V2

### 1.1 业务与数据特点

- 输入以 **Word（docx）** 为主：正文 + **大量复杂表格**（合并单元格、横向「上/下」分栏、多年份对比、多组「编号|测量值」横排、无表头或表头在合并格内等）。
- 目标输出仍为 **7 个业务字段** 的逐条记录（与现网契约一致，见 §4）。
- **格式不固定**：不存在「一张万能正则/一种固定模板」覆盖全库报告；但 **docx 内表格在 OOXML 中仍具有客观行列与单元格**。

### 1.2 V1 路径在强表格场景下的局限（设计层结论）

- 将表格 **仅** 展开为带 `|` 的线性文本，易弱化 **合并格、并排子表、列组边界** 等几何语义；**单元格底纹（红底超标）** 在纯 `cell.text` 路径中默认不可见。
- **整段长文本 + 一次性大 JSON** 易触发 **max_tokens 截断**，导致 JSON 无法解析或大量缺行。

### 1.3 V2 核心原则

| 原则 | 说明 |
|------|------|
| **解析与业务解耦** | 检修专用 **Word 摄入层** 独立实现，不强制复用 RAG 的 `DocumentParser` 作为 docx 表的主解析入口。 |
| **永不默认整篇进 LLM** | 以 **处理单元（Processing Unit）** 分块；单元内再 **按行批/子表** 限制单次模型输出规模。 |
| **结构化 ≠ 单一模板** | 「结构化」指 **保留表格网格事实 + 可编排的中间表示**，再配合 **多套启发式**；不是说全行业只有一种表格模板。 |
| **LLM 放在刀刃上** | 优先用程序生成「候选行」；LLM 主要负责 **语义对齐、缺陷/换管判定、歧义消解**，而非单次吐出上百行完整业务 JSON。 |

## 2. 总体架构（V2）

### 2.1 逻辑分层

```text
┌─────────────────────────────────────────────────────────────┐
│  接入层：上传 docx/pdf → 存储（如 MinIO）→ run 传入 URL/路径   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  文档摄入层（检修专用，独立于 RAG DocumentParser）               │
│  · docx：python-docx 遍历 body：段落 + 表格                   │
│  · 输出：章节上下文 + 表格矩阵（行列文本 + 可选合并/shading）    │
│  · pdf：文本层 / MinerU（与 V1 路由思想兼容，实现可新建模块）     │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  分块层：Processing Unit（禁止默认整篇）                       │
│  · 典型单元：「小节标题路径 + 紧邻段落块 + 其后表格列表」        │
│  · 超大表：按行窗口或按「逻辑子表」再切 batch                   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  候选记录层（混合）                                           │
│  A. 表格几何 → 行候选（管号、壁厚、列组归属）                  │
│  B. 多套启发式（上/下、2024/2023、编号|测量值横排…）           │
│  C. 候选不充分或置信低 → LLM（仅该单元、小批量、窄字段）         │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  业务规则与阈值：检测类型、缺陷类型映射、是否换管、阈值绑定       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  校验与输出：Pydantic、枚举、warnings、trace                   │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Word 摄入：为何以 python-docx 为主、Pandoc 为辅

| 方式 | 适用 | 说明 |
|------|------|------|
| **python-docx** | **主路径** | 直接访问 `w:tbl`、`w:tr`、`w:tc`，保留 **文档顺序**（段落与表穿插）；可在扩展中读取 **合并格 / shading**。 |
| **Pandoc** | **可选** | 适合生成统一中间格式或离线比对；复杂 Word 表合并行为需单独验证，**不宜单独作为生产唯一真理源**。 |

**结论**：V2 实现清单要求 **独立模块**（如 `inspection_docx` / `word_ingest`）封装 python-docx 遍历，**不得**把「必须先转成 Markdown」作为硬性前置。

## 3. 中间表示（建议）

### 3.1 `DocumentWalk`（示例命名）

- **段落序列**：`[{level?, text, style_hint?}]`
- **表格序列**：`[{table_id, rows[][{cells[]}], optional: merge_map, shading[][] }]`
- **章节锚点**：基于大纲序号、标题样式或正则（如 `^[一二三四五六七八九十]+、`）挂载 **当前小节路径**，供「检测位置」「行号」候选使用。

### 3.2 `ProcessingUnit`

- `unit_id`
- `heading_path`：如 `（一）炉膛水冷壁 / 1、B02吹灰器…`
- `paragraphs[]`：该单元叙事文本（原因分析、处理措施等）
- `tables[]`：指向摄入层表格 ID 及 **切片范围**（若一大表拆 batch）

### 3.3 `RowCandidate`（程序或 LLM 输出前的统一形状）

- `location_hint`、`row_hint`、`tube_raw`、`thickness_raw`
- `column_group`：如 `上|下`、`2024|2023`
- `flags`：`needs_llm`, `from_shading`, `parser_confidence`

## 4. 输出契约（与现网对齐）

单条记录必须满足：

| 字段 | 说明 |
|------|------|
| 检测位置 | 字符串，非空 |
| 行号 | 字符串，非空（可为「通用」「2-6」等语义化片段） |
| 管号 | **字符串**（支持 `-12`、`5-2` 等；禁止强行 number 导致精度丢失） |
| 壁厚 | 数值 |
| 检测类型 | `测厚` / `缺陷` |
| 缺陷类型 | 枚举之一；`测厚` 时必须为空 |
| 是否换管 | `是` / `否` |

响应外层建议仍为：`ok`、`records`、`summary`、`trace`（实现时可沿用现有 API 形态）。

---

## 5. 分块与 LLM 调用策略（硬性要求）

1. **禁止**：单次请求将 **整份 docx 文本** 作为唯一上下文让模型输出 **全部记录 JSON**。  
2. **必须**：  
   - 先划分 **Processing Unit**；  
   - 单元内 LLM 调用 **分批**（例如每批 ≤ 20～40 条候选行，可配置）；  
   - 输出格式优先 **窄字段**（仅填缺陷类型、是否换管、或对候选纠错），或使用 **NDJSON 每行一条** 以降低半截 JSON 风险。  
3. **max_tokens**：抽取阶段与分类阶段 **分开配置**；解析失败单元允许 **仅重试该单元**，禁止无限全文重试。

## 6. 表格语义：程序与 LLM 分工

### 6.1 程序侧（可靠部分）

- 从矩阵读出 **管号列 / 壁厚列** 的 **字符串与数值**（列索引可由启发式或轻量分类器确定）。
- **并排子表**：按列边界拆成多个逻辑子表（如左「上」右「下」、左 2024 右 2023）。
- **管号正负等业务规则**：可由确定性函数在候选生成后执行（规则来源：需求文档与 §8 规则摘要）。

### 6.2 LLM 侧（语义部分）

- **小节叙事 ↔ 表格绑定**（若单元边界算法遗漏）。
- **缺陷类型**、**是否换管** 与 **处理措施** 对齐。
- **启发式置信低**时的 **列角色识别**（哪列是管号、哪列是壁厚）。

### 6.3 样式通道（Phase 1 已纳入 docx）

- **docx**：摄入层读取单元格 **底纹填充（`w:shd/@w:fill`）**，命中可配置色系时标记为 **超标候选**，写入序列化文本供 LLM/规则消费。  
- **pdf / 扫描**：无底纹 XML 时仍依赖正文与阈值；可选 **表格区域截图 + 多模态**（按需开关，成本高），归入 Phase 2/3。

## 7. PDF / 扫描件

- 保留「文本层充足 → 文本解析；否则 MinerU」的 **路由思想**。  
- V2 实现可将 MinerU 输出映射到 **与 Word 相同的中间表示目标**（段落 + 表结构近似），再进入同一套 **Unit / RowCandidate** 流水线。

---

## 8. 业务规则摘要（实现时需与需求文档一致并可配置）

以下摘要仅供架构对齐，具体阈值与映射表以产品配置为准：

- **检测类型**：结合壁厚与区域阈值、叙事中的「超标/缺陷」等语义。  
- **缺陷类型**：封闭枚举；禁止模型发明新标签（失败则 `warnings` + 默认策略或人工复核队列）。  
- **是否换管**：结合「处理措施」「更换/保留」等；与 **检测类型** 一致性校验。  
- **管号正负（示例规则）**：若需求定义「上 → 负号、下 → 正号」等，应在 **确定性函数** 中实现，便于单测与审计。

## 9. API、配置与可观测性

### 9.1 API

- **同步（保留）**：`POST /inspection-extract/upload`、`POST /inspection-extract/run`；`run` 的 `content` 支持 MinIO URL；`source_type` 仍以 `docx`/`pdf` 等区分；**契约与行为不因异步能力而改变**。  
- **异步（可选，见 §15）**：`POST .../run/async` 返回 `job_id`；`GET .../jobs/{id}` 查状态与进度；`GET .../jobs/{id}/chunks` / `GET .../jobs/{id}/chunks/{work_idx}` **按块获取 parse 结果**，避免单次响应承载整份大 JSON。

### 9.2 配置（建议新增或独立前缀）

- `INSPECTION_V2_*` 或与检修模块共用前缀下增加：  
  - 每单元 LLM 最大批大小、parse/classify **分离**的 timeout、max_tokens  
  - 是否启用 shading、是否启用视觉兜底  
  - Word 摄入是否完全绕过旧 `DocumentParser`

### 9.3 可观测性

- **trace**：`unit_id`、`table_id`、`batch_index`、`ingest_parser_version`、`llm_stage`  
- **指标**：单元成功率、行级 salvage 率、校验失败原因分布  
- **日志**：中文关键节点日志（开始/结束单元、批次数、合并记录数），避免仅依赖截断后的 raw 日志判因

## 10. 风险与应对

| 风险 | 应对 |
|------|------|
| 表格极不规则，启发式覆盖不全 | 多策略并行 + 低置信走 LLM；记录 `warnings` |
| LLM 输出截断 | 小批量、NDJSON、提高 parse 专用 max_tokens、禁止单次超大数组 |
| 颜色语义丢失 | shading 读取或视觉通道（可选） |
| 与旧数据格式不一致 | API 契约保持稳定，内部替换管线 |

## 11. 分阶段实施计划（建议）

### Phase 1 — MVP（可验收）

- [ ] 独立 **`python-docx` 摄入**（与 RAG `DocumentParser` 扁平表并行存在）：段落 + 表格按行列展开 + **单元格底纹 → 超标候选**

- [ ] 合并单元格语义精细化（与 shading 已读的网格联动）

### Phase 3 — 优化

- [ ] PDF/MinerU 与 Word 中间表示对齐

- **新产品/新立项以 V2 为准**。  
- **V1** 仍可作为历史参考（MinerU 路由、Prompt 分层思想等），但 **Word 表解析主路径以 V2 为准**。

## 14. 后续优化（不在 Phase 1 范围）

### 14.1 异步长任务

当报告极大、多单元多批 LLM 导致 **同步 HTTP 耗时过长** 或网关超时风险升高时，启用 **§15 异步任务与断点续跑**（轮询按块拉取；可选 Webhook）。同步接口仍为默认主路径。

### 14.2 其它

- 多模态表格截图兜底、配置中心托管表格模板、人工抽检闭环等，仍按 Phase 2/3 推进。

## 15. 异步任务、分块增量返回与断点续跑（企业级目标架构）

> **目的**：在保留同步接口与现有 **parse 并发策略** 的前提下，支持 **按块尽早返回**、**任务持久化**、**进程/节点重启后续跑**，并与 **知识库摄入任务**（`IngestionOrchestrator` + `JobRepository` + 可选 `IngestionJobQueue`）采用同一套运维模型。下文为**简要、明确、全面**的落地约定。

### 15.1 四条约束（必须同时满足）

| # | 约束 | 设计要点 |
|---|------|----------|
| 1 | **每完成一块 LLM 解析即可对外可见** | 每块 parse **成功即写入侧挂文件**；客户端通过 **`GET .../chunks/{work_idx}`**（或等价）按块拉取，**不依赖**任务结束才下发整包大 JSON。终态若需整单汇总，可用单独 `GET .../result` 或 **预签名文件下载**，避免单次 HTTP 体过大。 |
| 2 | **持久化 + 断点续跑** | 任务元数据 **Job 级持久化**（见 §15.3）；分块结果 **块级文件**；重启后 **仅重跑未完成块**；classify/repair 前写入 **合并检查点**（`stage1_merged`），避免 classify 阶段崩溃后重复全量 parse。 |
| 3 | **保留现有同步接口** | **`POST .../run` 行为不变**，不强制经过异步任务状态机；同步路径仍可一次性返回完整 `InspectionExtractResponse`（与现网契约一致）。 |
| 4 | **保留当前并发处理策略** | Parse 阶段继续使用 **`INSPECT_EXTRACT_PARSE_CONCURRENCY`** + **`asyncio.Semaphore`**；异步路径仅 **跳过已持久化完成的块**，对 `pending` 块并发调度，**不调低并发、不改为串行**。 |

### 15.2 双通道架构

| 通道 | 说明 |
|------|------|
| **同步** | `extract_from_document` → `run_llm_extraction` 一次性完成 parse → 合并 → classify → repair → post_process；无 `job_id`、无任务目录落盘。 |
| **异步** | `POST .../run/async` 创建 `job_id`，后台执行；编排逻辑与同步 **复用同一套** `_parse_one_chunk`、classify、repair 与后处理，通过 **回调/钩子** 在每块完成时写侧挂文件并更新 Job 状态。 |

### 15.3 持久化（对齐知识库摄入 + 检修分块侧挂）

**任务级（与摄入一致的模式）**

- 独立 `state_dir`（如 `data/inspection_extract_jobs`），与 RAG `rag_jobs` **目录分离**。  
- **聚合快照**：`jobs.json`（或等价）**原子写**（`.tmp` → `replace`）。  
- **单任务记录**：`JobRepository.upsert/get`（本地 `jobs_index.json` 或 ES，与摄入同款能力）；字段含 `status`、`step`、`updated_at`、`metrics` 等；**全量 records 不塞进索引正文**，块级 bodies 走侧挂文件。  
- **可选 Redis**：复用 **`IngestionJobQueue` 形态**，更换 **`key_prefix`**（如 `inspection:extract:jobs`）；`enqueue/pop/ack`、`lease`、启动 **`requeue_processing_on_startup`**。  
- **启动恢复**：将不可靠的 `RUNNING` 转为可续跑状态并 **重新入队**；**`get_job`** 时 **持久化与内存按 `updated_at` 等合并**（同摄入 `get_job`）。  

**分块级（检修特有）**

- 路径示例：`{state_dir}/{job_id}/chunks/{work_idx}.json` — 存该块 **parse 输出的 records**（及必要元数据）。  
- `checkpoints/stage1_merged.json`：全部 parse 完成后合并排序，供 **classify 入口**；续跑时若已存在则 **跳过 parse，从 classify 起**。  
- **无表分块**：在参与 LLM 的列表中 **直接排除**，与产品定义一致，不落无效块文件。

### 15.4 对外 API（异步通道）

| 方法 | 职责 |
|------|------|
| `POST .../run/async` | Body 与同步 `run` 一致；立即返回 **`job_id`** 与轮询基址说明。 |
| `GET .../jobs/{job_id}` | **轻量**：`stage`、进度（如 `chunks_done/total`）、错误码；**不内嵌**全量 `records`。 |
| `GET .../jobs/{job_id}/chunks` | 各块状态（`pending/done/failed`、条数）。 |
| `GET .../jobs/{job_id}/chunks/{work_idx}` | 该块 **parse 完成后** 返回该块 JSON；未完成时约定 **404/202** 等（实现时固定一种并写进 OpenAPI）。 |

可选：`POST .../run/async` 带 **`callback_url`**（终态或分阶段 Webhook），用于 **B 端对接**；浏览器场景以 **轮询 + 按块 GET** 为主。

### 15.5 执行与续跑流程（简）

1. 提交 → 落 Job → `PENDING` → 入队（或进程内任务）。  
2. 文档解析与分块 → **过滤无表块** → 得到 `work_items` 并写入 `metrics`。  
3. **Parse**：`Semaphore(parse_concurrency)`；仅对 **无对应 `chunks/{i}.json`** 的块调 LLM；每完成一块 **先写文件再更新 Job**。  
4. 全块 **done** → 合并 → **`stage1_merged`** → classify → repair → 后处理 → **`COMPLETED`**。  
5. **重启**：扫描未完成 Job；parse 阶段 **跳过已有文件**；classify 之后可从 **`stage1_merged`** 重试后续。

### 15.6 实现分层建议

- **`InspectionExtractJobOrchestrator`（新）**：生命周期、`_save_state`、`JobRepository`、队列、恢复、`get_job`，**镜像** `IngestionOrchestrator`。  
- **`InspectionExtractService`**：同步入口不变；新增 **提交异步任务**、后台 `_run_job`。  
- **`InspectionExtractLlmOrchestrator`**：抽出 **可恢复 parse**（带 `on_chunk_done` 或等价）与同步路径共用，避免两套 LLM 规则。

### 15.7 验收要点

- 任一块完成后即可 **独立 GET** 该块结果，无需等大 JSON。  
- 重启后 **未完成块继续跑**、已完成块 **不重复请求** LLM。  
- 同步 **`/run`** 行为与契约 **不变**。  
- 日志/并发数可验证 **`parse_concurrency`** 仍生效。
