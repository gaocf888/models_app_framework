# API 接口配置与参数说明规范

## 1. 目标

为所有 `app/api/*.py` 接口提供统一的“可运维、可交接、可自解释”说明规范，确保开发者和运维在接口定义处即可明确：

- 接口依赖哪些服务与环境配置；
- 请求参数哪些必传、哪些可选；
- 不传可选参数时系统默认行为是什么。

## 2. 适用范围

- 适用于所有 FastAPI 路由文件（如 `rag_admin.py`、`chatbot.py`、`analysis.py`、`llm_inference.py`、`nl2sql.py`、`train_admin.py`、`small_model.py`）。
- 新增接口必须按本规范补充说明；存量接口按迭代逐步补齐。

## 3. 说明结构（两层）

每个 API 文件必须包含以下两层说明：

1) **模块级说明（文件头 docstring）**  
   说明该模块的前置依赖和关键配置项。

2) **接口级说明（函数 docstring）**  
   说明该接口参数规则与默认行为。

## 4. 模块级说明模板

建议模板：

```python
"""
<模块名称>接口。

服务配置前置条件（运维/开发）：
1) <依赖服务A>：
   - 需配置 <ENV_A>/<ENV_B> ...
2) <依赖服务B>：
   - 需确保 <连接/目录/权限> ...
3) 可选能力：
   - 若启用 <能力X>，需配置 <ENV_X> ...
"""
```

要求：

- 至少覆盖“核心依赖服务 + 核心环境变量”；
- 若有可选能力（如 RAG、Graph、Agentic），要明确开启条件和配置项；
- 描述要短，聚焦“能不能跑起来”。

## 5. 接口级说明模板

建议模板：

```python
"""
<接口功能一句话说明>。

参数说明（见 <RequestModel>）：
- 必传：a、b、c
- 可选：d、e、f
- 默认行为：<不传时使用的默认值/默认策略>
"""
```

要求：

- 参数命名必须与模型字段一致；
- 必传/可选必须明确，不写“等”；
- 默认行为必须可执行（如“默认 true”“默认 500/80/40”）；
- 若是 GET 且参数来自 query/path，也要写明来源（例如“query 参数”/“路径参数”）。

## 6. 参数文案规范

- 使用统一术语：
  - “必传” / “可选” / “默认行为”
  - “路径参数” / “query 参数” / “请求体”
- 描述风格：
  - 一行一个要点，避免长段落；
  - 不写实现细节代码名，优先写业务语义。

## 7. 配置项命名规范

- 环境变量统一使用反引号包裹，例如 `RAG_ES_HOSTS`；
- 配置项按“最小可用”优先列出，避免把全部参数一次性塞进说明；
- 推荐顺序：
  1. 存储连接
  2. 模型依赖
  3. 可选增强能力

## 8. 变更流程要求

当新增或修改接口时，必须同步完成：

1. 更新接口函数 docstring（参数与默认行为）；
2. 如新增依赖，更新模块头部“服务配置前置条件”；
3. 若影响对外能力，同步更新对应 framework-guide 文档章节。
