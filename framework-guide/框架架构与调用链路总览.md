# 框架架构与调用链路总览（给二次开发者）

> 目标：用一份简洁文档，让新同学在 10~20 分钟内搞清楚本工程的整体架构和主要业务调用链路，便于快速二次开发与接入业务。

---

## 1. 从架构角度看整体框架

### 1.1 分层概览

- **API 接入层（FastAPI）**
  - 入口文件：`app/main.py`
  - 对外暴露的主要接口：
    - 通用推理：`POST /llm/infer`
    - 智能客服：`POST /chatbot/chat`
    - 综合分析：`POST /analysis/run`
    - NL2SQL：`POST /nl2sql/query`
    - 小模型通道管理：`/small-model/...`
    - 大模型训练管理：`/train/...`
    - RAG 管理：`/rag/...`

- **应用编排 / Agent 层（LangChain + Agentic RAG）**
  - 主要组件：
    - `app/llm/chains/chatbot_chain.py`：Chatbot 多步链路（意图识别 + RAG + 回复生成）
    - `app/llm/chains/analysis_chain.py`：综合分析多步链路（任务规划 + RAG + 多模态汇总）
    - `app/nl2sql/chain.py`：NL2SQL 多步链路（问题理解 + 多命名空间 RAG + SQL 自检/修正）
    - `app/services/llm_inference_service.py`：通用推理服务（支持 Agentic 预处理 + Agentic RAG）
  - 公共能力：
    - `app/llm/prompt_registry.py`：Prompt 管理与 A/B 分流
    - `app/llm/langsmith_tracker.py`：LangSmith 可选埋点

- **RAG / 会话与上下文层**
  - `app/rag/embedding_service.py`：嵌入向量服务（sentence-transformers 企业级嵌入模型；离线优先/在线回退：`EMBEDDING_MODEL_PATH` 本地路径优先，否则按 `EMBEDDING_MODEL_NAME` 在线下载，失败时打日志并抛异常；依赖见 `requirements-大模型应用.txt`）
  - `app/rag/vector_store.py`：向量库抽象与内存实现（默认使用 FAISS 文件版索引，持久化目录由 `RAG_FAISS_INDEX_DIR` 控制）
  - `app/rag/rag_service.py`：通用向量检索 RAG 服务
  - `app/rag/hybrid_rag_service.py`：Hybrid RAG 服务（在 `RAGConfig.graph` 启用时，可根据配置在“仅向量 / 仅图 / 混合检索”之间切换）
  - `app/graph/ingestion.py` + `app/graph/query_service.py`：GraphRAG 图摄入与查询骨架（基于 Neo4j + LangChain Graph，是否启用由 `RAGConfig.graph.enabled` 控制）
  - `app/rag/agentic.py`：Agentic RAG 基座（`AgenticRAGService` + `RAGMode` + `RAGContext`）
  - `app/rag/ingestion.py` + `app/api/rag_admin.py`：RAG 知识摄入与数据集管理（在 GraphRAG 启用时，会在向量写入的同时调用 `GraphIngestionService` 写图）
  - `app/conversation/store.py` + `app/conversation/manager.py`：会话存储（内存/Redis）与上下文管理

- **NL2SQL 子系统**
  - `app/nl2sql/schema_service.py`：Schema 元数据同步（支持 SQLAlchemy 连接真实 DB）
  - `app/nl2sql/rag_service.py`：NL2SQL 专用 RAG 命名空间（schema / biz / qa）
  - `app/nl2sql/prompt_builder.py`：构建 NL2SQL Prompt
  - `app/nl2sql/validator.py`：SQL 安全校验（只读/只 SELECT）
  - `app/nl2sql/executor.py`：SQL 执行骨架（后续接入多数据源）
  - `app/nl2sql/chain.py`：NL2SQL 多步链路（规划 + RAG + 生成 + 自检）
  - `app/services/nl2sql_service.py`：封装链路并对接 API

- **小模型与训练管理层**
  - 小模型推理：`app/small_models/channel_manager.py`、`workers.py`、`inference_engine.py`
  - 小模型训练：`app/small_models/training.py` + TensorBoard 集成
  - 大模型训练：`app/train/llm_training.py` + `llm_factory_adapter.py` + `train/orchestrator.py`

- **核心基础设施**
  - 配置：`app/core/config.py`（统一配置中心）
  - 日志：`app/core/logging.py`
  - 指标：`app/core/metrics.py`（Prometheus 指标）

> 若需要架构更详细说明，可参考：`docs/大小模型应用技术架构与实现方案.md`。  
> **RAG / GraphRAG 实现细节、配置项与架构图**见：`framework-guide/RAG整体实现技术说明.md`。

## 2. 从业务角度看调用链路（如何“走通一条线”）

下面从四个最常用的业务能力出发，说明一次完整调用从 HTTP 到模型的链路，方便二次开发者快速定位修改点。

### 2.1 通用推理 `/llm/infer`

**典型调用场景**：统一入口，做单轮/多轮问答，可选择是否启用 RAG/上下文，以及 Agentic 预处理。

1. **API 层**
   - 文件：`app/api/llm_inference.py`
   - 路由：`POST /llm/infer`
   - 请求模型：`app/models/llm.py` 中的 `LLMInferenceRequest`
     - 关键字段：
       - `user_id` / `session_id`
       - `prompt` 或 `messages`
       - `model`（逻辑模型名）
       - `prompt_version`
       - `enable_rag` / `enable_context`
       - `rag_mode`（`basic` / `agentic`，可选）

2. **Service 层**
   - 文件：`app/services/llm_inference_service.py`
   - 核心方法：`LLMInferenceService.infer(req)`
   - 主流程（简化版）：
     1. 记录用户消息到 `ConversationManager`
     2. 若 `rag_mode="agentic"` 且 LangChain 可用 → `_analyze_intent_and_plan(...)` 做问题诊断与规划
     3. 若启用 RAG → 调用 `AgenticRAGService.retrieve(...)` 获取上下文片段
     4. 根据 `PromptTemplateRegistry` 选择 Prompt 模板
     5. 通过 LangChain ChatOpenAI（或回退到 `VLLMHttpClient`）调用模型
     6. 记录助手回复到会话，回写 `LLMInferenceResponse`

3. **向下依赖**
   - 配置：`LLMConfigRegistry` → 读默认模型与模型参数
   - RAG：`AgenticRAGService` → `RAGService` → `EmbeddingService` + `VectorStoreProvider`
   - 会话：`ConversationManager` → 内存/Redis Store

> 二次开发建议：  
> - 如果只是新增一种“推理场景”，通常在 Prompt 层（`prompts.yaml` + `PromptTemplateRegistry`）和少量业务逻辑上增加分支即可；  
> - 若要扩展 Agentic 能力，可在 `LLMInferenceService` 中增加新的 Planner / Tool 步骤，但注意保持 basic 模式兼容。

---

### 2.2 智能客服 `/chatbot/chat`

**典型调用场景**：面向终端用户的问答/客服机器人，可按需接入 RAG 与多轮对话。

1. **API 层**
   - 文件：`app/api/chatbot.py`
   - 路由：`POST /chatbot/chat`
   - 请求/响应模型：`app/models/chatbot.py` 中的 `ChatRequest` / `ChatResponse`

2. **Service 层**
   - 文件：`app/services/chatbot_service.py`
   - 核心方法：`ChatbotService.chat(req)`
   - 主逻辑：
     - 优先尝试初始化并使用 `ChatbotChain`（LangChain 版智能客服链路）；
     - 如依赖缺失，则回退到占位实现（不走大模型，仅返回简单文本）。

3. **链路层（多步 Workflow）**
   - 文件：`app/llm/chains/chatbot_chain.py`
   - 核心方法：`ChatbotChain.run(user_id, session_id, query, ...)`
   - 主流程（简化版）：
     1. 根据 scene=`chatbot` + `user_id` 从 `PromptTemplateRegistry` 选择 Prompt 模板
     2. 调用 `_analyze_intent(...)` 进行意图识别与路由规划（通过 LangChain LLM），并把结果以隐藏 SystemMessage 注入
     3. 读取最近会话历史，拼接上下文
     4. 若启用 RAG → 调用 `AgenticRAGService.retrieve(...)` 获取知识片段，并以 SystemMessage 拼接
     5. 最终通过 LangChain ChatOpenAI 生成回答，并通过 `LangSmithTracker` 记录 trace

> 二次开发建议：  
> - 新增业务意图或路由策略：可优先修改 `_analyze_intent(...)` 的 Prompt 或增加简单规则；  
> - 若要接入新的知识源，只需在 RAG 层增加对应命名空间和摄入逻辑，然后在 Chatbot 场景中使用合适的 `scene` / `namespace` 即可。

### 2.3 综合分析 `/analysis/run`

**典型调用场景**：输入一段分析需求 + 多模态数据引用，输出结构化分析报告。

1. **API 层**
   - 文件：`app/api/analysis.py`
   - 路由：`POST /analysis/run`
   - 请求/响应模型：`app/models/analysis.py` 中的 `AnalysisInput` / `AnalysisResult`

2. **Service 层**
   - 文件：`app/services/analysis_service.py`
   - 核心方法：`AnalysisService.run_analysis(data)`
   - 主逻辑：
     - 记录用户请求到会话；
     - 若 LangChain 可用 → 调用 `AnalysisChain` 完成多步分析；
     - 否则回退到占位实现，给出简单的占位报告。

3. **链路层（多步 Workflow）**
   - 文件：`app/llm/chains/analysis_chain.py`
   - 核心方法：`AnalysisChain.run(data, prompt_version=None)`
   - 主流程（简化版）：
     1. `_plan_analysis(...)`：使用 LLM 输出分析任务规划（目标 + 子任务 + 证据类型）
     2. 根据 scene=`analysis` 获取 Prompt 模板，注入规划结果 SystemMessage
     3. 如启用上下文 → 读取历史对话并拼接
     4. 如启用 RAG → 通过 `AgenticRAGService` 检索与当前分析相关的知识片段
     5. 添加多模态数据概览 SystemMessage
     6. 以用户需求作为最终 HumanMessage，调用 LLM 生成分析报告

> 二次开发建议：  
> - 若要增加更复杂的多步流程（如真正按子任务循环 RAG），可以在 `AnalysisChain` 内扩展 `_plan_analysis` 与 RAG 调用逻辑；  
> - 若要接入图像/视频/传感器小模型工具，可在“多模态占位”部分增加对小模型服务的调用，并将结果以额外 SystemMessage 注入。

### 2.4 NL2SQL `/nl2sql/query`

**典型调用场景**：将自然语言问题转换为安全的 SQL（只读 SELECT），并可对接真实业务数据库。

1. **API 层**
   - 文件：`app/api/nl2sql.py`
   - 路由：`POST /nl2sql/query`
   - 请求/响应模型：`app/models/nl2sql.py`

2. **Service 层**
   - 文件：`app/services/nl2sql_service.py`
   - 核心方法：`NL2SQLService.query(req)`
   - 主逻辑：
     - 调用 `NL2SQLChain.generate_sql(question, user_id)` 生成 SQL；
     - 通过 `SQLExecutor` 执行 SQL（当前为骨架，可对接实际 DB）；
     - 使用 `ConversationManager` 记录 NL2SQL 查询历史（可用于上下文）。

3. **链路层（多步 Workflow）**
   - 文件：`app/nl2sql/chain.py`
   - 核心方法：`NL2SQLChain.generate_sql(question, user_id=None)`
   - 主流程（简化版）：
     1. `_plan(...)`（如 LangChain 可用）：理解问题，输出候选表/实体、关键字段、是否需要 join/聚合等规划摘要
     2. 基于规划结果调用 `NL2SQLRAGService.retrieve(...)`：
        - 从 `nl2sql_schema` / `nl2sql_biz_knowledge` / `nl2sql_qa_examples` 三个命名空间联合检索上下文片段
     3. 使用 `PromptBuilder` 组合 System + Schema/Biz/QA 片段 + 用户问题构建 NL2SQL Prompt
     4. 通过 LangChain ChatOpenAI 或 `VLLMHttpClient` 生成 SQL 初稿
     5. 使用 `SQLValidator` 做安全校验：
        - 通过 → 返回；
        - 不通过且 LangChain 可用 → 调用 `_refine_sql(...)` 再生成一次仅含安全 SELECT 的 SQL，并再次校验；
        - 若仍不通过 → 返回空字符串。

> 二次开发建议：  
> - 要接入真实库：优先配置 `DatabaseConfig.url`，并用 `SchemaMetadataService.refresh_from_db()` 同步 Schema；  
> - 要增强 NL2SQL 准确率：重点在 PromptBuilder、规划 `_plan(...)` 与 RAG 命名空间的数据质量上做文章。

## 3. 二次开发者的推荐入手路径

- **只想“用”这个框架接业务**：
  1. 先读：`docs/大小模型应用技术架构与实现方案.md` 的第 2、4 章；
  2. 再看本文件第 2 章对应你关心的业务（推理 / 分析 / Chatbot / NL2SQL）；  
  3. 按需要修改对应 Service/Chain 的业务逻辑和 Prompt。

- **想深入改 Agent / RAG / NL2SQL 策略**：
  1. 在 `docs/Agentic-Workflow-设计蓝图.md` 中对照各场景的 Workflow 设计；
  2. 根据第 6 章“提升方向”决定是先做配置化、还是先补测试/评估；
  3. 按场景修改对应 Chain 和 RAG 层代码，并同步更新 `memory-bank/05-progress-log.md`。

- **想快速找到某个链路相关代码**（记忆小抄）：
  - 通用推理：`/llm/infer` → `llm_inference.py` → `LLMInferenceService`
  - Chatbot：`/chatbot/chat` → `chatbot_service.py` → `ChatbotChain`
  - 综合分析：`/analysis/run` → `analysis_service.py` → `AnalysisChain`
  - NL2SQL：`/nl2sql/query` → `nl2sql_service.py` → `NL2SQLChain` + `NL2SQLRAGService`

> 若在二次开发过程中遇到“应该改哪里”的问题，可以先根据上面的链路提示定位模块，再参考 `memory-bank/05-progress-log.md` 看该模块的历史变更与设计背景。这样可以最大程度避免与现有架构冲突。

## 4. Docker 部署与数据持久化一览

- 本框架在容器环境中部署时，涉及以下**关键持久化点**：
  - 会话与上下文：依赖外部 Redis（通过 `REDIS_URL` 连接，由 Redis 自身的数据目录挂盘）；
  - RAG 向量库：`RAG_FAISS_INDEX_DIR` 目录下的 `faiss.index` + `faiss_meta.json`，需通过卷挂载实现容器外持久化；
  - 训练产物与模型权重：大模型/小模型训练输出目录与 TensorBoard 日志目录，建议统一到 `/data/...` 路径并挂盘；
  - 应用文件日志（如启用）：应将日志目录挂载为持久卷，或尽量输出到 stdout/stderr。
- 详细的挂载示例与模块清单，见：`framework-guide/数据持久化与容器部署说明.md`。
