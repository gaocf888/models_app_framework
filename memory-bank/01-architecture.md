# 系统架构与技术决策（大小模型统一基座）

## 1. 总体架构视图

从全局视角看，系统划分为以下几个层次（自上而下）：

1. **接入层（API / Gateway）**
   - 基于 FastAPI 的异步服务，提供：
     - 大模型推理、智能客服、综合分析、NL2SQL 的 HTTP / WebSocket 接口。
     - 小模型通道管理与状态查询接口。
   - 统一接入鉴权、速率限制、Trace ID 注入。

2. **应用编排层（LangChain / Agent 层）**
   - 利用 LangChain/LangGraph 对大模型相关任务进行编排：
     - 聊天客服链路（带 RAG/上下文）。
     - 综合分析 Agent 链路（多模态输入 + RAG + 工具调用）。
     - NL2SQL Pipeline（RAG + Prompt 编排 + LLM + SQL 校验/修正）。
   - 提供统一的「链」与「Agent」注册与路由机制。

3. **模型服务层**
   - **大模型推理服务**：通过 vLLM 部署的 LLM/多模态模型，采用 HTTP/gRPC 接口。
   - **大模型训练/微调服务**：
     - 可视化：LLaMA-Factory 前端 + 后端；
     - 代码：内部训练脚本封装，支持 LoRA/全参微调、断点续训等。
   - **小模型推理服务**：
     - 通道管理（Channel Manager）；
     - 视频解码线程、任务队列、有界消息队列、推理线程；
     - 多算法配置化（YOLO 等），支持模型热加载或按通道配置。
   - **小模型训练服务**：
     - 代码训练脚本；
     - TensorBoard/其他可视化面板集成。

4. **RAG / 上下文 / 会话管理层**
   - RAG 管理：
     - 文档/Schema/多模态特征的向量化与存储；
     - 支持传统 RAG（单轮检索 + 生成）和 Agentic RAG（多步推理 + 工具链）。
   - 会话与上下文：
     - 基于 Redis 的会话存储，支持多用户多会话；
     - 会话摘要与裁剪策略；配置化控制是否启用上下文。

5. **数据与基础设施层**
   - Redis、业务数据库、向量数据库、对象存储（文件/模型）、配置中心。
   - 监控与日志（Prometheus + Grafana + Loki）。

## 2. 关键设计原则

- **大小模型架构区分但在接入与监控层统一**
  - 小模型：视频流通道、解码-队列-推理的流水线式架构，重视线程安全与资源生命周期管理。
  - 大模型：请求 → 任务队列 → Worker Pool → vLLM 调用，重视请求级并发与 LangChain 调用层性能。

- **LangChain 作为编排层，而非所有逻辑的承载层**
  - 模型调用、RAG、Agent、提示词与 A/B 测试在 LangChain 层统一封装；
  - 但重计算/IO 密集逻辑（如大规模文件读写、长耗时小模型推理）保持在独立服务中，通过工具/接口方式被 Agent 调用。

- **可配置、可观测、可迭代**
  - RAG 开关、上下文是否启用、不同 Prompt 版本、A/B 测试分流策略全部配置化；
  - 与 LangSmith、Prometheus/Grafana 集成，实现链路级与系统级监控；
  - 所有实验（Prompt/模型版本/A/B 测试）可记录并回溯。

- **线程安全与资源释放优先**
  - 小模型按通道锁、全局对象锁、stop_event + 有界队列；避免 `queue.empty()` 等不可靠原语。
  - 大模型 Worker Pool 与任务队列不依赖 empty()；LangChain 实例设计成无共享可变状态、配置只读。

## 3. 设计模式与结构性决策

### 3.1 大模型侧

- **调用模式**：FastAPI（异步） → 应用编排层（LangChain Chain/Agent）→ vLLM 推理服务。
- **线程模型**：
  - API 层使用 `async`/`await` 与 Uvicorn/Gunicorn workers（多进程）；
  - LangChain 部分：
    - 避免在全局单例中持有有状态的链实例；
    - 将链定义为可重复构造的「工厂」，每次请求基于不可变配置创建轻量实例（或使用每 worker 级缓存）。
  - vLLM 本身内部已经优化高并发，LangChain 调用需尽量减少额外序列化/反序列化与阻塞操作。

- **Prompt 版本与 A/B 测试**：
  - 使用「策略注册中心」+ 配置（数据库/配置文件）实现：
    - 按功能模块（客服/综合分析/NL2SQL）维护多版本 Prompt 模板；
    - 按比例路由到不同 Prompt/模型组合实现 A/B 测试；
    - LangSmith 集成记录每次调用的策略版本与效果指标。

- **Agentic RAG 设计**：
  - 使用 LangChain/LangGraph：
    - 将 RAG 检索、推理调用、小模型工具调用（例如调用检测服务）包装为 Agent 可用工具；
    - 支持多步推理（先问再查、再合并分析）。

### 3.2 小模型侧

- 采用在《高并发多线程安全应用架构设计》中定义的**子方案 A**：
  - 通道管理器（Channel Manager）负责：
    - `channelId → ChannelContext` 的线程安全持久映射；
    - 每通道锁、stop_event、解码/推理 Future 的生命周期。
  - 单通道内部：
    - 解码线程（生产者）→ 有界 SPSC 队列 → 推理线程（消费者）。
    - 停止与收尾通过 `stop_event` + `get(timeout)` 实现，不依赖 `empty()`。
  - API 层：
    - FastAPI 的路由负责 start/stop/update 通道；
    - 对同一通道请求在 per-channel 锁下串行执行。

- 多算法与配置化：
  - 通道上下文中包含算法配置（模型类型、权重路径、预处理配置等），由通道创建或更新时加载；
  - 支持在不重启服务的前提下切换模型。

### 3.3 RAG、上下文与会话管理

- **RAG 策略**：
  - 传统 RAG：检索 Top-K 文本/Schema → 拼接 Prompt → 单次生成。
  - Agentic RAG：Agent 按需多次检索、调用多种知识源（文档、图像特征、传感器数据摘要等）。
  - 两种方式都通过统一的 RAG 配置中心管理：
    - 是否启用 RAG；
    - 使用的向量库、检索策略；
    - 不同业务域/场景的命名空间划分。

- **会话与上下文**：
  - Redis 中存储：
    - 用户会话元数据（user_id, session_id, last_active 等）；
    - 上下文摘要（压缩过的历史对话、NL2SQL 历史查询摘要等）；
    - 大模型记忆（用于长期个性化时）。
  - LangChain 部分可复用其 Memory 抽象，但底层实现绑定 Redis。

### 3.4 NL2SQL 子系统集成

- 按《NL2SQL系统概要设计》采用三层架构：
  - 接入层：FastAPI 路由 → NL2SQL 应用服务。
  - 智能层：Schema RAG + Prompt 编排器 + LLM 调用器 + SQL 校验与自我修正。
  - 数据访问层：多数据源适配、SQL 执行与安全控制。

- 在本基座中：
  - LLM 调用器、RAG 与会话管理与其他大模型能力共享。
  - NL2SQL 作为一个独立的 LangChain「链」或 Agent 工具接入整体编排框架。

## 4. 性能与可用性考虑

- **大模型高并发**：
  - FastAPI + Uvicorn workers，多进程模型；
  - 请求层适度限流，未命中缓存/未启用流式的请求可使用任务队列降峰；
  - vLLM 端启用批处理（batching）与 KV cache，LangChain 调用时尽量使用流式响应（减少尾延迟）。

- **LangChain 调用层不成为瓶颈**：
  - 避免每次请求都重新构建大量 Python 对象或进行昂贵的 IO；
  - 根据不同业务场景，将链配置缓存在线程本地或 worker 级别；
  - 对公共 Prompt 模板采用编译/预渲染（如 Jinja2 模板 + 变量填充）方式。

- **小模型多线程与高可用**：
  - 严格遵循「全局对象锁 + per-channel 锁 + 有界队列 + stop_event」方案；
  - 出错时通道级隔离，避免单一路径异常影响其他通道；
  - 关键资源（视频流、GPU/推理会话）绑定通道生命周期，确保释放顺序：stop_event → 队列收尾 → 注销通道。

## 5. 监控与可观测性

- **指标（Prometheus）**：
  - API 级：QPS、响应时间分布、错误率、状态码分布。
  - 大模型：LLM 调用耗时、token 数量、vLLM 服务可用性、调用失败率。
  - 小模型：每通道帧率、队列长度、丢帧数、推理耗时。
  - RAG：检索耗时、命中数、命中质量评分（可选）。
  - NL2SQL：生成 SQL 成功率、执行失败率、平均执行时间。

- **日志（Loki）**：
  - 统一结构化日志（JSON）：
    - `trace_id`, `span_id`, `user_id`, `session_id`, `module`, `level`, `message`, `extra`。
  - 对关键链路记录：
    - 请求参数摘要（脱敏）、Prompt 版本、RAG 命中摘要、模型响应摘要等。

- **可视化（Grafana）**：
  - 仪表盘：
    - 系统总览（CPU/Mem/Disk/Network）。
    - 大模型服务性能与错误率。
    - 小模型通道健康状态。
    - RAG/向量库性能。

## 6. 对后续详细设计的约束

- 所有模块在详细设计与落地时应遵循：
  - 不破坏大小模型架构区分与统一编排层这一前提；
  - 新增能力（如新的 Agent、工具或模型）必须以组件形式接入 `02-components.md` 中定义的组件体系；
  - 所有接口暴露前需定义清晰的输入/输出与错误语义；
  - 所有长期运行的线程/进程管理遵从《高并发多线程安全应用架构设计》的约束。
