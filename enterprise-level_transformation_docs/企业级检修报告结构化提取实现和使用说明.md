## 2. 对照检查结论（先给结论）

### 2.1 是否按主文档实现

结论：**核心链路已实现，且 Phase 1 + Phase 2 的主体能力已落地。**

已实现项：
1. 独立模块与独立接口：`/inspection-extract/upload`、`/inspection-extract/run`。
2. 两段式生产调用：先上传到 MinIO，再传 URL 到 run。
3. 文档解析路由：`docx/pdf` 支持；`pdf` 扫描件可自动走 MinerU。
4. LLM 三阶段抽取：`parse` → `classify` → `repair`。
5. 规则后处理：枚举归一、缺陷类型映射、检测类型判定、是否换管归一。
6. 阈值绑定增强：位置匹配 + 段落近邻优先 + 谨慎全局回退。
7. 可追溯 warnings：记录阈值绑定值与命中来源。
8. 监控指标、配置项、测试覆盖已补齐。

### 2.2 是否达到企业级生产要求

结论：**达到“可生产试运行（Beta）”水位。**

已具备：
1. 与智能客服/综合分析同级的模块化接入。
2. 结构化输出契约 + Pydantic 强校验。
3. 对象存储化上传链路（MinIO）与远程 URL 解析。
4. 阈值规则可解释（命中来源可追溯）。
5. 指标与日志可用于排障与质量追踪。

建议继续优化（Phase 3）：
1. 场景化模板（机组/锅炉型式分版）。
2. 更细粒度段落块与表格块作用域绑定。
3. 引入人工修正闭环数据做提示词迭代。

## 4. 业务流程与实现细节

### 4.1 端到端流程（生产调用）

```mermaid
flowchart LR
  A[调用方上传文档] --> B[/inspection-extract/upload]
  B --> C[MinIO 持久化 + 预签名 URL]
  C --> D[调用 /inspection-extract/run]
  D --> E[文档解析路由 docx/pdf/mineru]
  E --> F[LLM parse]
  F --> G[LLM classify]
  G --> H[LLM repair]
  H --> I[规则后处理 + 阈值绑定]
  I --> J[Pydantic 校验 + 输出 records/summary/trace]
```

### 4.2 解析路由

- `source_type=docx/doc`：优先 `DocumentParser`（可保留 Word 表格顺序）。
- `source_type=pdf`：
  - 文本层充足：普通 PDF 解析；
  - 扫描件：MinerU 路由解析为 markdown 再抽取。
- `content` 支持 URL（如 MinIO 预签名链接）：
  - 服务端先下载为临时文件，再进入上述解析流程。

### 4.3 结构化字段与规则判定

必填字段：
- 检测位置、行号、管号、壁厚、检测类型、缺陷类型、是否换管

约束规则：
1. `检测类型` 只能是 `测厚` / `缺陷`。
2. `缺陷类型` 仅允许 8 类枚举值。
3. 当 `检测类型=测厚` 时，`缺陷类型` 必须为空。
4. 当 `检测类型=缺陷` 且缺陷类型缺失时，按规则补齐并写告警。

### 4.4 阈值绑定（当前增强版）

阈值来源从报告文本中抽取（如“低于3.15mm超标”），并按以下优先级匹配：
1. **段落近邻**（位置语句与阈值语句在近邻行窗口内）
2. **位置强匹配**（`hint in location`）
3. **位置相似匹配**（token 重叠评分）
4. **全局回退**（仅当文档中只有一个阈值规则）

每条记录会写入：
- `阈值绑定:<x.xxx>mm`
- `阈值命中来源:<段落近邻/位置强匹配/位置相似匹配/全局回退/未命中>`

## 5. 接口使用说明

### 5.1 上传接口

- 路径：`POST /inspection-extract/upload`
- 请求：`multipart/form-data`，字段 `file`

响应示例：

```json
{
  "ok": true,
  "file_name": "2024检修报告.docx",
  "object_name": "inspection_extract/xxxx_2024检修报告.docx",
  "source_type": "docx",
  "url": "https://minio.xxx/presigned-url",
  "bucket": "chatbot-images"
}
```

### 5.2 运行接口

- 路径：`POST /inspection-extract/run`
- 请求模型：`InspectionExtractRequest`

最小示例：

```json
{
  "user_id": "u_demo",
  "session_id": "s_demo",
  "source_type": "docx",
  "content": "https://minio.xxx/presigned-url",
  "strict": false,
  "return_evidence": true,
  "prompt_version": "v1"
}
```

响应关键字段：
- `records[]`：结构化明细
- `summary`：总数/缺陷数/换管数/warnings
- `trace`：解析路径、模型、模板版本、耗时

## 6. 提示词与配置说明

### 6.1 提示词（`configs/prompts.yaml`）

- `inspection_extract`：兜底模板
- `inspection_extract_parse`：候选记录抽取
- `inspection_extract_classify`：分类判定
- `inspection_extract_repair`：结构修复

建议：
1. 保持“仅输出 JSON”硬约束；
2. 枚举白名单写死在模板中；
3. 版本变更时优先保留旧版做灰度。

### 6.2 环境变量（`InspectionExtractConfig`）

| 环境变量 | 默认值 | 作用 |
|---|---|---|
| `INSPECT_EXTRACT_ENABLED` | `true` | 模块开关 |
| `INSPECT_EXTRACT_STRICT_DEFAULT` | `false` | 默认 strict 行为 |
| `INSPECT_EXTRACT_MAX_REPAIR_RETRIES` | `1` | 结构修复重试次数 |
| `INSPECT_EXTRACT_PROMPT_VERSION` | `v1` | 默认提示词版本 |
| `INSPECT_EXTRACT_MODEL_NAME` | 空 | 指定模型（为空回退 `LLM_DEFAULT_MODEL`） |

MinIO 复用配置（chatbot image）：
- `CHATBOT_IMAGE_MINIO_ENDPOINT`
- `CHATBOT_IMAGE_MINIO_ACCESS_KEY`
- `CHATBOT_IMAGE_MINIO_SECRET_KEY`
- `CHATBOT_IMAGE_MINIO_BUCKET`
- `CHATBOT_IMAGE_MINIO_SECURE`
- `CHATBOT_IMAGE_MINIO_AUTO_CREATE_BUCKET`
- `CHATBOT_IMAGE_MINIO_PRESIGN_TTL_SECONDS`

## 7. 监控与运维

指标（`app/core/metrics.py`）：
1. `inspect_extract_requests_total{status}`
2. `inspect_extract_parse_latency_seconds`
3. `inspect_extract_llm_latency_seconds`
4. `inspect_extract_records_total`
5. `inspect_extract_validation_fail_total`

运维建议：
1. 监控 `failed` 请求与 `validation_fail` 增长趋势；
2. 抽样分析 `warnings` 中“阈值命中来源”分布；
3. 针对扫描 PDF 关注 MinerU 耗时与失败率。

## 8. 企业级验收清单（建议）

上线前至少完成：
1. upload/run 端到端联调通过（含 MinIO 预签名 URL）。
2. docx、文本层 pdf、扫描 pdf 三类样本验证通过。
3. 必填字段完整率、枚举合法率达标。
4. 多阈值样本下位置错绑率低于目标阈值（建议人工抽检）。
5. 指标、日志、错误码可用于排障闭环。

## 9. 风险与后续路线

P0（当前）：
1. upload -> run 两段式已落地；
2. 阈值绑定支持段落近邻与位置匹配；
3. warnings 可追溯判定来源。

1. 引入异步任务化（大文件时避免同步接口超时）；
2. 阈值作用域进一步细化到章节块/表格块；
3. 增加规则版本化与回滚开关。

P2（中期）：

2. 人工修正样本驱动提示词优化；
3. 可选多模态视觉增强图片页理解。
