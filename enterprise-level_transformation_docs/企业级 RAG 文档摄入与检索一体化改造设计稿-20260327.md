# 企业级 RAG 文档摄入与检索一体化改造设计稿（2026-03-27）

## 1. 目标与范围

- 建立统一的 Knowledge Ingestion Platform（知识摄入平台）。
- 支持多源文档：文本、Markdown、HTML、PDF、Docx（可扩展 OCR）。
- 摄入结果统一写入：
  - 向量索引（ES/EasySearch dense vector）
  - 全文索引（ES/EasySearch text/BM25）
  - 元数据索引（版本、权限、来源、质量指标）
- 检索统一升级为：多路召回 + 融合 + 重排 + 策略化路由。
- 形成可观测、可回滚、可审计、可灰度的生产运维能力。

## 2. 总体架构（逻辑分层）

### A. 接入层（API）

新增/重构 `app/api/rag_admin.py` 下接口族：

- `POST /rag/jobs/ingest`：提交摄入任务（异步）
- `GET /rag/jobs/{job_id}`：查询任务状态
- `POST /rag/jobs/{job_id}/retry`：失败重试
- `POST /rag/documents/upsert`：同步小文档快速通道（可选）
- `POST /rag/documents/delete`：按文档/版本删除
- `POST /rag/query`：调试查询（线上服务检索走业务链路）

说明：现有 `/ingest/texts`、`/ingest/raw_document` 归为兼容接口，逐步弱化。

### B. 编排层（Ingestion Orchestrator）

新增 `app/rag/ingestion_orchestrator.py`：

- 任务状态机：`PENDING -> RUNNING -> SUCCESS | FAILED | PARTIAL`
- 步骤编排：
  1. source load
  2. parse
  3. normalize
  4. chunk
  5. enrich(metadata/entity/acl)
  6. embed + index
  7. quality check
  8. finalize(alias/version)
- 支持幂等键：`tenant + namespace + doc_name + doc_version`

### C. 文档处理层（Pipeline）

重构 `app/rag/document_pipeline.py` 为多阶段可插拔：

- `parsers/`：`pdf_parser.py`, `html_parser.py`, `md_parser.py`, `docx_parser.py`
- `cleaners/`：去页眉页脚、去目录噪音、重复段合并、编码修复
- `splitters/`：
  - `structure_splitter`（按标题层级）
  - `semantic_splitter`（句边界/段落语义）
  - `window_splitter`（滑窗兜底）
- `enrichers/`：
  - chunk hash
  - section path
  - source page
  - ACL 标签
  - language/domain tags

### D. 存储层（Index + Metadata）

在现有 `ElasticsearchVectorStore` 基础上升级：

- 三类索引（可同库不同 index）：
  1. `rag_chunks_v{n}`：chunk 主索引（text + vector + metadata）
  2. `rag_docs_v{n}`：文档元数据索引（doc-level）
  3. `rag_jobs_v{n}`：任务索引（状态、日志摘要、错误）
- alias 管理：
  - `rag_chunks_current`
  - `rag_docs_current`
  - `rag_jobs_current`
- migration 工具：
  - `app/rag/migrations/index_migrator.py`
  - 支持 create version index / backfill / alias switch / rollback

### E. 检索层（Unified Retrieval）

重构 `app/rag/rag_service.py` 为策略驱动：

- 召回：
  - dense recall（vector）
  - lexical recall（BM25）
  - metadata recall（可选）
- 融合：
  - RRF + 可调权重
- 重排：
  - CrossEncoder（`BAAI/bge-reranker-large`）
- 输出：
  - 标准 `RetrievedChunk`（含来源、分数、section、doc_version）

- `doc_version`
- `namespace`
- `tenant_id`
- `source_uri`
- `checksum`
- `status`
- `created_at` / `updated_at`

### Chunk（分块级）

- `chunk_id`
- `doc_id`
- `chunk_index`
- `text`
- `embedding`
- `section_path`
- `page_no`
- `chunk_hash`
- `acl_tags`
- `pipeline_version`

- `target_doc(s)`
- `status`
- `step`
- `error_code` / `error_message`

## 5. 与其他模块联动改造范围（必须同步）

### GraphRAG

- 摄入后钩子从 `RAGIngestionService` 改为 orchestrator 的 `post_index_hook`。
- 同步 `doc_version`，避免图与索引版本错位。
- 删除/回滚时同步图侧清理策略。

### NL2SQL

- `NL2SQLRAGService` 改为读取标准 `RetrievedChunk`（带 section/doc 信息）。
- 命名空间策略独立：schema/biz/qa 使用差异化 chunk 策略。
- 支持 schema 文档增量 upsert。

### 通用推理 `/llm/infer`

- 继续走 `AgenticRAGService`，但结果对象改为标准结构（便于 trace）。
- scene profile 保留，按场景调参。

### Chatbot / Analysis / Agentic

- 统一接入 `RetrievalPolicy`（而非硬编码参数）。
- 支持 Agentic 多步中的子问题检索计划。

## 6. 任务状态与错误处理（轻量增强版）

采用“企业可用但不过载”的状态模型：

- 状态：`PENDING/RUNNING/SUCCESS/FAILED/PARTIAL`
- step 失败处理：
  - 记录结构化错误日志（含 doc_id, step, error_code）
  - 写入 job 状态
  - 管理接口返回明确错误
- retry 模式：
  - 从失败 step 重试
  - 全量重试

说明：不引入复杂补偿编排引擎，但满足运维排障和重跑。
