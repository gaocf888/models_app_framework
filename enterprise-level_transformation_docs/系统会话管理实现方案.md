# 系统会话管理实现方案

> **文档目的**：说明本框架中「多轮会话上下文」的存储、索引、HTTP 接口与跨模块使用方式，供架构评审、联调与二次开发查阅。  
> **代码主路径**：`app/conversation/`、`app/api/chatbot.py`（会话运维 API）、各业务 `Service` 中的 `ConversationManager` 注入与调用。  
> **当前实现态（已落地）**：Redis 热层 + EasySearch 冷层（写入即归档）+ 查询自动回查冷层 + 对象存储备份增强（可配置）。

---

## 1. 目标与范围

### 1.1 目标

| 目标 | 说明 |
|------|------|
| **统一会话抽象** | 智能客服、通用 LLM 推理、综合分析、NL2SQL 等复用同一套「追加消息 / 读历史 / 清理」接口。 |
| **可横向扩展** | 生产环境通过 **Redis** 在多 Worker、多实例间共享会话；开发可用内存实现。 |
| **可运维** | 支持 TTL、单会话消息条数上限、按用户列举会话目录（左侧栏类场景）、按会话导出/删除。 |
| **可长期保存** | 会话消息在热层写入后旁路归档到 EasySearch 冷层；热层过期后仍可通过回查获得历史。 |
| **可增强备份** | 冷层归档同时支持对象存储备份（`local` / `s3`），用于容灾与低成本长期留存。 |
| **可治理** | `user_id` / `session_id` 强校验、Redis 故障时不「假成功」、连接与超时可配置。 |

### 1.2 范围与非目标

- **范围内**：算法侧会话消息与目录（方案 B：Redis ZSET + Hash + List）+ EasySearch 冷层归档/回查 + 对象存储备份增强；与业务用户登录、BFF 用户表无绑定。
- **范围外**：调用方自有「会话分组 / 置顶 / 产品级会话表」；若需完全由业务库维护目录，可参考 `docs/会话列表与语义标题-落地方案.md` 中的方案 A。

## 2. 核心概念

### 2.1 会话维度

- 会话由二元组 **`(user_id, session_id)`** 唯一确定。
- **`user_id`、`session_id` 均由调用方传入**；框架不签发用户身份，仅与 **Service API Key**（`Authorization: Bearer`）配合使用。
- **Redis key 布局依赖上述字符串拼接**，故默认 **禁止 ID 中含冒号 `:`**（见第 5 节）。

### 2.2 分层结构

```
业务 API（chatbot / llm / analysis / nl2sql）
        ↓
ConversationManager（统一入口：校验 ID + 热层写入 + 冷层归档/回查编排）
        ↓
热层：ConversationStore（内存）或 RedisConversationStore（Redis + 独立 IO 线程事件循环）
        ↘
冷层：ConversationArchiveStore（EasySearch 消息索引 + 会话汇总索引）
        ↘
备份层：Object Backup（local jsonl 或 s3 对象存储）
```

| `app/conversation/store.py` — `get_default_store()` | **进程内单例**：所有未显式传入 `store` 的 `ConversationManager` 共享同一后端与（Redis 时）同一连接池 / IO 线程。 |

| `app/main.py` | 注册 `ConversationIdValidationError` → HTTP **422**。 |
| `app/models/chatbot.py` 等 | 请求体字段 `user_id` / `session_id` 的 Pydantic `field_validator` 与 `ids` 模块联动。 |

## 4. 数据模型

### 4.1 消息条目（List 元素 / 内存 dict）

每条消息建议字段：

- `role`：`user` / `assistant`（及可能的 `system`）
- `content`：文本正文（当前实现中，`user` 消息在持久化时可能附带 `[image_urls]` 块用于历史回显）
- `ts`：写入时间戳（秒，浮点）

补充（智能客服图片场景）：
- `GET /chatbot/sessions/messages` 返回层会将 `content` 中的 `[image_urls]` 附加块解析为独立 `image_urls: []` 字段；响应里的 `content` 仅保留用户文本正文。

### 4.2 Redis 键设计（方案 B）

| Redis Key 模式 | 类型 | 用途 |
|----------------|------|------|
| `conv:{user_id}:{session_id}` | **List** | 消息序列；元素为 JSON 字符串。 |
| `conv:index:{user_id}` | **Sorted Set** | 会话目录索引；member=`session_id`，score=**最近活跃时间（毫秒）**。 |
| `conv:meta:{user_id}:{session_id}` | **Hash** | `title`、`title_source`、`first_user_preview`、`message_count`、`title_updated_at` 等。 |

### 4.3 内存实现对应关系

- 消息：`{(user_id, session_id): [messages]}`
- 目录：`_catalog_meta[user_id][session_id]`，字段与 Redis Hash 对齐，供 `GET /chatbot/sessions` 使用。

### 4.4 冷层索引（EasySearch）

| 索引 | 用途 | 关键字段 |
|------|------|----------|
| `conversation_messages_v1`（默认） | 消息归档与会话消息回查 | `message_id`、`user_id`、`session_id`、`role`、`content`、`ts_ms`、`title_snapshot`、`title_source_snapshot`、`archived_at_ms` |
| `conversation_sessions_v1`（默认） | 会话目录冷层回查与标题快照 | `session_doc_id`、`user_id`、`session_id`、`last_activity_at_ms`、`title`、`title_source`、`message_count` |

### 4.5 标题与列表展示

- **首条 user 消息**写入时，根据 `CHATBOT_SESSION_TITLE_MODE` 生成 `title` / `first_user_preview`（`truncate` / `off`；`llm` 预留与 `truncate` 行为一致时可查代码）。
- 列表展示优先级见 `session_catalog.display_title`：**Hash.title → preview 截断 → session_id 缩写**。
- **`GET /chatbot/sessions/messages`**：响应体含 `title`、`title_source`（`get_session_title_snapshot`），与列表同行一致。
- **`PATCH /chatbot/sessions/title`**：请求体 `{"title":"..."}`，查询参数 `user_id`、`session_id`；写入 meta 的 `title` 与 `title_source=user`（`session_catalog.normalize_edited_title`，长度上限 `CHATBOT_SESSION_TITLE_EDIT_MAX_RUNES`）。

## 5. 标识符校验（企业级）

- 实现：`app/conversation/ids.py`。
- 规则摘要：
  - 去首尾空白；非空；最大长度见 `CONV_USER_ID_MAX_LEN`、`CONV_SESSION_ID_MAX_LEN`。
  - 禁止控制字符（正则过滤）。
  - **`CONV_ID_STRICT=true`（默认）禁止 `:`**，避免与 `conv:{user_id}:{session_id}` 拼接产生歧义。
- 异常：**`ConversationIdValidationError`**  
  - HTTP：在 `main.py` 中统一映射为 **422**，body 形如 `{"detail": "..."}`。  
  - 请求体模型：`ChatRequest`、`LLMInferenceRequest`、`AnalysisInput`、`NL2SQLQueryRequest` 等对 `user_id`/`session_id` 做 `field_validator`，在入参阶段即失败。

## 6. 行为与策略

### 6.1 TTL 与会话过期

- 环境变量：`CONV_SESSION_TTL_MINUTES`（分钟，0 或负数表示按实现可视为不过期）。
- **内存**：`get_recent_history` 命中过期会 `clear` 并返回空；`list_sessions` 会同步剔除过期项。
- **Redis**：对消息 List、index ZSET、meta Hash 在写入链路上 **`EXPIRE`**（与 TTL 一致）；列表接口对「List 已过期但 ZSET 仍残留」的成员做 **惰性 ZREM + DEL meta**。
- **冷层回查**：当热层会话因 TTL 过期后，若启用 `CONV_QUERY_FALLBACK_COLD=true`，仍可从 EasySearch 返回历史与目录。

### 6.2 消息条数上限

- **`CONV_MAX_HISTORY_MESSAGES`**：单会话在存储中的最大条数；超出时 **裁剪保留最近 N 条**（内存裁剪 + Redis `LTRIM`）。
- **`CONV_EXPORT_MAX_MESSAGES`**：`get_session_messages` / 管理读全量的上限（与上下文窗口用的 `get_recent_history` 上限可不同）。

### 6.3 Redis 写入语义

- 默认 **`append_message` 异步**（`run_coroutine_threadsafe` + 回调记错）。
- **`CONV_REDIS_APPEND_WAIT_MS > 0`**：写入后阻塞等待 Future 完成（超时即记错误日志），用于收紧「写完立刻读」的竞态窗口。
- **读失败**（`get_recent_history` / `get_messages`）：返回 **空列表**并打 error 日志，**不回退**到进程内空字典（避免误判「无历史」）。
- **`clear` 失败**：不打成功路径的本地清理，避免误以为已删。

### 6.4 归档与回查语义（当前实现）

- **写入即归档**：`append_user_message` / `append_assistant_message` 在热层写入后调用 `ConversationArchiveStore.archive_message`。
- **幂等键**：`message_id = sha256(user_id + session_id + role + ts + content)`；重复写入同一消息会落到同一文档 ID。
- **失败隔离**：归档失败仅记录日志，不影响在线问答主链路。
- **目录回查**：`list_sessions` 先读热层；热层不足时补查冷层并按 `session_id` 去重后统一排序分页。
- **消息回查**：`get_session_messages` 合并冷/热结果，按 `(role, content, ts)` 去重并按 `ts` 升序返回。
- **标题回查**：`get_session_title_snapshot` 热层无标题时回查冷层；`update_session_title` 成功后同步更新冷层会话索引。

### 6.5 Redis 客户端

- `REDIS_SOCKET_CONNECT_TIMEOUT`、`REDIS_SOCKET_TIMEOUT`（秒）。
- `REDIS_HEALTH_CHECK_INTERVAL`：若当前 `redis-py` 不支持该参数，实现中会捕获 `TypeError` 后降级重连（无 health check）。

## 7. HTTP 接口（智能客服）

> 业务路由需 **`Authorization: Bearer <SERVICE_API_KEY>`**（环境变量 `SERVICE_API_KEYS` / `SERVICE_API_KEY`）。

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/chatbot/sessions` | Query：`user_id`（必填）、`limit`、`offset`、`order`；返回会话目录与展示标题等。 |
| `GET` | `/chatbot/sessions/messages` | Query：`user_id`、`session_id`、可选 `limit`；导出/历史消息。 |
| `DELETE` | `/chatbot/sessions/messages` | Query：`user_id`、`session_id`；删除消息 List + 索引 + meta（Redis）或内存等价清理。 |

**重要**：`app/api/chatbot.py` 中 **`ChatbotService` 与上述运维接口共用同一 `ConversationManager` 实例**，保证对话写入与列表/删除读写一致。其他模块各自 `ConversationManager()` 仍通过 **单例 `get_default_store()`** 共享同一 Redis 命名空间。

`/chatbot/sessions/messages` 响应约定补充：
- `messages[].content`：纯文本内容；
- `messages[].image_urls`：图片链接数组（仅 `role=user` 可能非空）。

## 8. 跨模块使用与键冲突防范

以下模块通过 **`ConversationManager`** 读写会话（默认注入或自建 `ConversationManager()`）：

- `ChatbotService` / `ChatbotChain` / `ChatbotLangGraphRunner`
- `LLMInferenceService`
- `AnalysisService` / `AnalysisGraphRunner`
- `NL2SQLService`

**Redis 下共用 `conv:{user_id}:{session_id}`**。若 Chatbot 与 NL2SQL 使用相同的 `user_id` + `session_id`，**消息会在同一 List 混写**。企业集成建议：

- 为不同产品线使用 **`session_id` 前缀或命名空间**（例如 `chat-uuid`、`nl2sql-uuid`）。

## 11. 故障与排查提示

| 现象 | 可能原因 |
|------|----------|
| 列表为空但有对话 | 旧数据仅有 List 无 ZSET；需新消息触发索引写入，或评估迁移脚本。 |
| 写完立刻读不到最后一条 | Redis 异步写入窗口；可调大 `CONV_REDIS_APPEND_WAIT_MS`。 |
| 422 与 ID 相关 | ID 含 `:` 或控制字符、超长；检查 `CONV_ID_STRICT` 与客户端传参。 |
| Redis 错误后历史变空 | 当前设计为 **失败返回 []** 而非静默回退；查应用日志与 Redis 连通性。 |
| 多业务消息串话 | 相同 `user_id`+`session_id` 被多模块共用；改为不同 `session_id` 前缀。 |
| 热层过期后查不到历史 | 检查 `CONV_ARCHIVE_ENABLED` 与 `CONV_QUERY_FALLBACK_COLD` 是否开启、冷层索引是否可读。 |
| 对象备份无文件/对象 | 检查 `CONV_ARCHIVE_OBJECT_ENABLED`、`CONV_ARCHIVE_OBJECT_BACKEND` 与 local 目录权限 / s3 凭据。 |
